# 软件工程正处于临界点

## 📝 内容总结

**演讲主题**：通过"软件生态学"视角，探讨AI对开发者生态系统的影响。

**核心要点**：

**1. 软件生态学的定义**
软件生态学是整体性研究产生软件的"社会-技术"生态系统，包含相互依存的开发者、工具、组织及环境，强调万物互联与共同进化。

**2. 关键理论框架**
- **系统思维**：系统的各要素相互关联，环境本身也是系统的一部分
- **复杂适应系统（CAS）**：具备成长、变化、进化与涌现特性，难以完全预测
- **社会-技术系统**：人与技术共同构成的系统，其运作方式深受组织结构与文化影响

**3. 康威定律的启示**
组织构建的技术反映其内部沟通结构；同时，组织的价值观与文化同样深刻影响开发者生态的演化方向。

**4. 谷歌实践案例**
作为深度工程驱动的组织，谷歌在文化层面强调透明度、互帮互助、无指责复盘、可持续性优于英雄主义；在技术层面采用单体代码仓库、基于主干开发、统一构建工具链与全球测试平台等高标准化方案，以支撑庞大生态的高效运转。

**5. AI时代的挑战与应对**
演讲深入分析了在AI驱动开发场景下，代码量10倍增长对测试基础设施、版本控制系统、代码审查流程、发布策略、内部API安全、token管理等各个环节带来的系统性冲击，强调理解开发者生态系统的整体性是把握未来发展方向的关键。

【音乐】>> 大家好，你们好吗？嘿，挺好的，不错。周三下午三点，对吧？今天是周三，还不错。欢迎来到《软件工程：站在临界点》这个节目。我叫Adam Bender，今天我想和大家聊一个叫做"软件生态学"的概念，这个词你们可能之前没听过。我想通过它来聊聊AI对我们开发者生态系统的影响，就是你们每天都在用的那个开发者生态。

不知道你们有没有注意到，现在外面的世界变得挺魔幻的，对吧？你2026年的工作绝对不是你2020年想象的那个样子。我敢保证。如果你试图跟2020年的我描述现在的情况，他肯定不会相信。要弄清楚这些变化会把我们带向何方，确实有点让人头大，对吧？虽然我无法预测未来，但我认为，如果我们仔细研究当下的软件生态系统，我们想要的答案其实比我们想象中近得多。我觉得"软件生态学"这个概念，正好能给我们一个完美的视角，来审视行业的这个特殊时刻。

我能听到你们在问了："软件生态学是什么？他是不是为了上台演讲临时编的这个词？"我没有瞎编，这是个真实的概念。但在我下定义之前，我想先给你们铺点背景。所以，我们要稍微深入聊聊"系统思维"，希望你们能接受。跟着我走，等讲到那儿你们就全明白了。

我们先从"系统"这个概念说起。一个系统，就是一组相互关联的要素，这些要素按照一定规则运作，从而形成一个统一的整体。听着挺高大上，但其实你们都懂系统。系统长这样——这玩意儿你们今天应该挺感激它的，就是空调，对吧？有个温控器知道该设多少度，有个暖通设备来调节温度，还有个房间。温度合适了，信号就停了。这就是系统。如果你是软件工程师，你应该天天都在跟系统打交道。你设计系统、构建系统、运营系统。但跟系统打交道这么久，你肯定学到了一件事：万物互联。一切都是相连的。这个主题我今天会反复强调几次。

接着说说"生态系统"，它是系统的一种。这个定义比较长，耐心听一下：生态系统是一个动态的、相互依存的行动者网络，这些行动者会与环境共同进化，具有涌现行为和去中心化自主性等特征。生态系统很复杂，它们由复杂的组件构成。这些组件之间深度相连，它们有自主性，能做决策、能行动。对我们来说最关键的是，环境本身就是这个系统的一部分。没错，环境是系统的一部分，你没法把两者分开。所以，生态系统也是"复杂适应系统"的一种。

复杂适应系统（CAS）的特点就是它能随着时间成长、变化和进化。所有酷炫的系统都是复杂适应系统，对吧？复杂适应系统还有"涌现"这个特性。涌现性是指，你光看系统的任何一个单独部分都看不出来的东西，只有把整个系统拼起来，才能看到那种自然而然冒出来的行为。正是这种不断的变化、学习，再加上涌现，让我们很难搞明白生态系统里到底在发生什么。

提到生态系统，或者说在你们走进这个会场之前想到生态系统，你脑海中浮现的可能是这个画面。说实话，我现在倒挺想待在那儿的，看着挺惬意的。但你的内部开发环境，也是一种生态系统。你有各种工具和服务，有复杂的参与者，有一群有想法、有目标要完成的人，还有各种业务约束。这也是一个系统。事实上，它是一种特殊的系统，叫做"社会-技术系统"。

社会-技术系统说白了就是个高级说法，指由人和技术共同构成的系统。社会-技术系统超级复杂，复杂的原因是，你先把那些技术堆在一起，然后把人掺进去，对吧？不过，你们很可能在不知不觉中接触过一些关于社会-技术系统的智慧。你们有没有听说过"康威定律"？肯定都听过对吧，简单说一下：组织构建的技术，会反映出其内部的沟通结构。通俗点讲，如果你让四个团队一起写编译器，那你最后会得到一个四遍扫描的编译器。事情就是这样。

说到底，康威定律揭示了一个观察：我们构建技术的方式，和构建它的组织结构是分不开的。组织决定了会造出什么东西。当然啦，影响开发者生态的不只是组织结构，组织的价值观和文化同样影响深远。你的生态会构建出组织激励你去造的东西。你的工程文化会塑造开发者生态的整个环境。

你看，一旦你了解了社会-技术系统，你会在软件开发领域到处看到它们的身影。从架构到事后复盘文化，从代码审查到安全策略，无处不在。我们造的东西，以及我们选择造东西的方式，都是我们价值观的映射。只要我们用心，就可以用这些认知来放大我们的价值观，把它们融入我们构建的东西里，从而推动我们想要的结果。

所以现在，我觉得我可以正式给你们定义"软件生态学"了。软件生态学是…… 以下是这段内容的中文口语化翻译：

整体性地研究产生软件的"社会-技术"生态系统。好，现在大家都跟上节奏了。如果刚才那些内容听起来有点抽象，没关系，因为接下来我们要看一个真实的例子。我们要聊聊谷歌的开发者生态系统。谷歌的开发者生态系统非常庞大，老实说，也相当复杂，因为在过去25年里，我们发展出了真正了不起的能力——没错，也带来了一些复杂性。首先声明一下，我之所以拿谷歌来说事儿，部分原因是因为我在那儿工作，但它确实是我最了解的开发者生态系统。我今天的目的不是要告诉你"你必须照搬谷歌的做法"。那对你来说未必合适，对吧？你们是不同的公司，处在不同的位置，面临的是不同的取舍。

和所有工程组织一样，谷歌做了很多选择——这些选择都是基于我们当时建设生态系统的具体需求。而你们的情况？不一样。完全不一样。

几年前，我们写了一本书。也许你们中有人见过这本书。内部呢，我们叫它"火烈鸟书"。我们写这本书的目的，是想让人们了解谷歌内部的实际运作方式。书的一半内容讲的是版本控制、测试之类的东西，但整整前半本讲的都是**工程文化**。我们当时收到很多质疑：为什么要花这么多篇幅讲文化？答案很简单——如果你不了解谷歌的文化，你就无法理解我们为什么会做出那些技术选择。这两者是紧密关联的。

如果你没读过这本书也没关系，我给你简单过一遍。别担心，你不用全读完。拍照也行，没问题。谷歌有几个独特的地方：

比如说，我们是**深度工程驱动**的——做重要决策时，工程师一定在场。我们**非常强调透明度**——尽可能把所有的文档和代码都开放出来。我们鼓励大家**互帮互助**。事实上，你问任何从谷歌离职的人，他们最怀念的往往是同事之间的那种互助精神。我们把**代码评审**看作指导别人的机会，而不是打分的考试。我们**极其重视标准化**，真的非常重视。我们相信**持续改进**。我们非常推崇**无指责的事后复盘**（blameless postmortems）。我们坚信**可持续性胜过英雄主义，自动化胜过重复劳动**。

当然，这些理想我们也不总是能做到，但这就是我们在文化层面追求的目标。

技术层面呢，想必你们都听说过我们的**单体代码仓库**（monolithic repository），我一会儿细说。我们采用**基于主干开发**（trunk-based development），每次变更都直接落到主干。在谷歌构建一个二进制文件时，几乎每一行代码都是从源码编译出来的——这挺疯狂的吧？我们有**统一的构建工具链**，所有人都在用。我们有**全球测试自动化平台**，一个地方跑所有的测试——每天几十亿次测试。我们有一个**全局的"最后一次绿灯"信号**，我只要看一个简单的内部网站，就能知道任何构建的状态。我们有**统一的计算环境**，所以"在我这儿能跑"这种情况基本不存在。我们有**有自己主张的开发框架**，以及**一小撮核心编程语言**。

正是这种**文化与技术的混合体**，造就了今天的谷歌。你不能只理解其中一半而忽略另一半。

如果非要我挑出**一个原则**——一个似乎一直在（哪怕是无意识地）指引我们的原则——那就是**"共同命运"**（shared fate）这个概念。

所谓"共同命运"，描述的是一个生态系统及其各个组件之间紧密相连的程度。在一个高度共同命运的生态系统里，一个组件可以影响所有其他组件。

在开发者生态系统中，"共同命运"既是一个技术选择，也是一个社会选择。你不能仅仅通过强制大家使用相同的技术来实现共同命运——你还得建立相应的社会契约，来约定大家如何管理这些技术。

在谷歌，我们的"共同命运"始于那个**单体代码仓库**。谷歌的每一行代码——除了Android和Chrome等少数例外——都在**同一个代码仓库**里。所有的代码。所有的。一切都提交到主干。没有分支。没有版本。所有东西都在一个地方。

这种"共同命运"让我们能做到一件很牛的事：在一个文件里打一个安全补丁，然后**一周之内，全公司所有的应用就都修好了**。这就像拥有超能力一样，对吧？**10行代码放对地方，就能修补100亿行的应用和系统软件**。相当厉害吧？

当然，"共同命运"也不总是好事。它本质上是一个**取舍**。有些场景下，"共同命运"并不合适。比如在我们的生产环境中，我们不想让一个服务把其他所有服务都拖垮，也不想让一个集群影响到整个区域。所以谷歌下了很大功夫——特别是为了避免那些会引发**级联故障**的危险类型的"共同命运"。

重点是：**"共同命运"是一个权衡**。你得想清楚在哪些地方采用它，然后确保它对你来说是合适的。

在我们大规模的"共同命运"环境里，涌现出的最有趣的一个特性，就是所谓的**大规模代码变更**（large-scale changes）。这个能力**远在AI出现之前就有了**——在AI之前很久，谷歌的内部工具就已经能让一个开发者改动**几百万行代码**。几百万行。他们可能永远不会去看那些代码，可能再也不会见到，可能根本不认识那些代码的所有者…… 关于这件事，对吧？我们今天已经构建了一些工具，让这一切能够自动实现。而且我们用这种方式已经做了至少15年了。正是这种能力让我们能够随着时间推移不断发展我们的单一代码库（mono repo）。我们可以更新语言和框架。这真的是防止我们内部环境陷入停滞的关键。可以毫不夸张地说，如果没有它，就不会有今天的谷歌。对吧？从事这些工具工作的人会告诉你，这就像是无名英雄一样的工作，让公司能够以需要的速度前进。

大规模变更（large-scale changes）这件事的特点是，除非你理解让这一切成为可能的文化组件和技术部分，否则你无法真正领会它的价值。比如，你需要一种广泛普及的测试文化。每个人都必须写测试，我才能做这件事。我们需要一个统一的平台，这样我就知道去哪里获取这些测试的结果。对吧？我们需要有通用的构建工具，这样大家构建出来的软件才是一样的。我构建，你构建，我们得到相同的结果。我们需要一套标准的库，这样在替换东西的时候，我们就不用到处跑，去找哪个版本的库对你能用、对我不能用，对吧？我们需要标准化的代码审查。我们需要在单一代码库本身里有透明度，这样我才知道哪些代码需要修改。

LSC（大规模变更）真正是我们谷歌"用自动化取代苦活"理念的终极体现，而这一切之所以可能，是因为整个生态系统的存在。你不能指着我们开发者环境的某一个部分说："这就是LSC能发生的原因。"而是所有东西都连接在一起的。对吧？

现在，每一个开发者生态系统——你的、你工作过的——都有这些涌现出来的特性（emergent properties）。这些通常是让你觉得"挺有我们公司特色"的东西。这是因为它们通常是你为了塑造工作方式而做的一系列选择的组合结果。所以，每个生态系统都有类似的东西。谷歌有大规樢变更，你有别的东西。谷歌的开发者生态系统嵌入在我们特定的工程文化中，产生了一套独特的权衡（trade-offs）。这些权衡服务于我们的技术和业务目标。

然而，谷歌的生态系统，和所有生态系统一样，不可能所有事情都擅长。我们选择了为极致的规模、安全性、性能做优化，对吧？而且我们通常愿意付出开发者生产力的代价来换取这些。这是我们生态系统中愿意做的权衡。

其他地方——比如五个人初创公司——的开发者生态系统看起来会不一样，这也没关系。在一个五人初创公司里，速度和敏捷性可能是你能拥有的最重要的东西。介于五个人初创公司和谷歌之间的某个地方，就是你们所在的地方，对吧？你们大多数人所在的生态系统规模介于5到20万之间。你们要做的权衡真的非常值得关注，因为当你审视这些权衡的时候，你就能开始理解这个组织的价值观。它真正在乎什么？不是嘴上说在乎什么，而是你的组织真正在乎什么？当你理解了这些价值观，你就能开始在转型展开的过程中塑造它。

我们现在都在经历大量的转型，所以理解我们要去哪里会很有帮助。

到现在为止，"软件生态系统"这个概念应该更清晰一些了。我认为现在该谈谈房间里那头吃token的大象了。一个AI优先的开发者生态系统到底是什么样的？

要是能从零开始构建一个这样的生态系统——全新的一片绿地——当然很好，但我敢打赌你们都没有这个奢侈的条件，对吧？你们所有人都必须在继续交付软件、继续做现在正在做的事情的同时，替换掉它字面上每一个部分。没什么大不了，对吧？你的公司依赖你继续交付价值，并确保没有东西出问题。

所以，为了确保不出问题，让我问你一个问题：你对你现在的开发者生态系统理解得有多深？对吧？你能把它全部画出来吗？你知道所有部分在哪里吗？不仅仅是技术部分，还有社会部分。你能列出你的生态系统实际上是由什么构成的吗？你的组织里其他人对它的理解程度如何？它的优点和弱点是什么？你的瓶颈在哪里？你在哪些地方受到约束，哪些地方是自由的？你有什么样的可选性？你自己的生态系统中见过什么样的涌现特性？

也许最重要的是，如果你的生态系统突然需要在未来18个月内增长10到15倍，你知道什么会先崩溃吗？

你看，最后这个问题的关键在于，我们所有人都得很快回答它。因为地球上每一个开发者生态系统都在经历一场剧烈的转型。也许它还没扩散到你所在的每个地方，但它正在到来。每一个开发者生态系统都将不得不面对这个10倍时刻的想法。我真希望能告诉你有办法绕过它，但它正在到来，对吧？

这是令人激动的时代，但也是有点让人困惑的时代。过去25年里我们精心演变出来的所有权衡，都将要被重新平衡。你们可能从这里的每位演讲者那里都听到过类似的话。我们还不知道未来会是什么样子。我们正在努力搞清楚。一些人预测生产力会达到10倍、100倍。我们以数量级来衡量。这很多，对吧？突然之间，10倍增长的问题不再只是一个思想实验。对你和你公司来说，这就像是一个紧急红色警报。你将不得不去想办法解决它。如果不是今天，那肯定在未来12个月内。

现在，在我继续讲下去之前，我真的想强调一点：我理解在…… 生成代码的速度快了10倍，工程交付的速度快了10倍。这是两码事。其实在谷歌我们经常说，工程就是编程在时间维度上的积累。问题是，我们正在大幅加快编程速度。我们让"写代码"这台机器跑得飞快。所以，我们得想办法围绕这台代码机器重新设计工程流程，才能真正把成果交付给客户。没人知道这种生产力增长能推到什么程度。我是真不知道。也许下周就停了呢。应该不会吧。但有一点是肯定的，我们只会越走越远。问题在于，我觉得我们还有大量工作要做，因为我敢打赌，我们现在做事的方式在10倍规模下是行不通的。我敢说你们现在构建软件的方式、我现在交付软件的方式，在10倍甚至100倍的速度下是玩不转的。有些东西必须得变。

咱们先从标准的开发者生态说起吧。我知道这简化过了，但希望你们能认出其中一些概念，不过我还是用你们更熟悉的形式来呈现。是一张挺复杂的图，对吧？在一个生产力或者说活动量提升10倍的世界里，什么必须得变？咱们先从跟"把代码写进代码库"相关的节点说起。顺便说一下，我把测试和版本控制暂时排除掉了，等下再讲它们。

如果那些绿色节点突然要做10倍的工作，会怎样？咱们先从写源代码说起。如果每个人写代码都快多了，那代码量大概率也会暴增，对吧？这可不是什么好事。现在特别适合重温一下杰夫·阿特伍德那句老话：软件是负债。一上来就10倍的代码、10倍的负债，局面已经不太妙了。顺便提一句，你大概也发现了，你不能光给每个人发一堆token就指望他们自求多福。所以希望你们在投资对员工的再培训。你们在投资再培训，对吧？你们知道所有工程实践都文档化在哪了吗？要是必须演进的话，你们知道怎么改吗？回去好好想想这些问题吧。

咱们快速过一下构建系统。至少，代码更多意味着编译时间更长，对吧？系统更大，编译时间更长。这可真"雪中送炭"。我敢肯定你们公司从来没人抱怨过编译慢。猜怎么着？编译只会更慢。要是很多工作都是Agent在驱动，那就意味着编译次数也更多。不光是代码更大，而且是次数更多。编译既花时间又费算力。你可能从来没注意过自己在编译上花了多少时间，但我可以保证，代码量变成10倍的时候，你一定会感受到的。对吧？

那所有这些代码的设计呢？你有没有合适的Agent技能来鼓励解耦？有没有合适的服务器框架来确保你能既快又安全地组合各种能力？说到这儿，你们公司现在到底有多少种Web应用的服务方式？实际在跑的服务器框架到底有多少种？我反正不知道。如果Agent在写所有这些代码，你打算怎么管理组件复用？也许你赌这事不重要。但要是Agent写出来的代码写起来容易、维护起来难，你也别惊讶。这好像已经是当前的水平了。Agent很擅长写代码，但它们不一定总能像你我一样做长远考虑。那代码，肯定不会整理得很好。没关系，那部分我们能解决，但事实是Agent现在帮我们干了太多活。我们得每天琢磨怎么才能最有效地用好它。

搞不好某天，所有这些Agent生成的代码会让你们的二进制文件大到根本编译不动。或者大到根本没法装进手机里发出去。你们有没有问过这个问题：你能编译的最大的二进制文件是多大？我其实也不知道答案，但我知道谷歌已经在撞墙了。有些地方我们的二进制文件已经大到编译不动了。肯定能解决，我有信心，但事实是，"大"会带来一大堆后果。规模到处都在产生影响。

也许你们是微服务那一派的，正在看着我嘀咕："我干嘛要建那么大的服务？我所有服务都很小。"那挺好，但我现在有个问题：10倍的网络流量会怎样？10倍的服务、10倍的通信量、10倍的网络流量，你们准备好了吗？没人能毫发无损地躲过去。规模的影响无处不在。

假设你没法稳定地编译所有这些代码。我相信你肯定能，对吧？那你的代码审查流程会变成什么样？现在大家都很担心代码审查，我觉得这担心是有道理的。我们正在给这个高度依赖人力的流程施加巨大压力。对吧？而且在很多情况下，它正在变成瓶颈。我对人有一个了解：没人想当瓶颈。一旦你给他们施压，他们的反应就会很奇怪，对吧？

代码量变成10倍，你会遇到两种情况中的一种：要不就是改动量变成10倍大，要不就是改动数量变成10倍多。那对你们的代码审查人员意味着什么呢？我敢打赌，你们大多数的技术负责人根本撑不住那种审查速度，哪怕一天带5个这种10倍效率的开发者都费劲，对吧？工作量太大了。所以，为了不当瓶颈，他们会怎么做？他们会开始重新调整流程。会开始抄近道，确保自己不挡任何人的路，因为谁都不想当拦路虎。

也许你能在一定程度上用AI来解决这个问题，我承认。也许你能部署AI让审查流程更高效。好吧？但那只能解决一部分问题。因为如果你们团队的人不写……（内容到此中断） 写代码，但这是他们唯一会遇到它的时间，就是在代码审查的时候。但他们在审查时并没有那么专注。这正是我们刚才说的。那么，谁在关注代码库的变化呢？嗯，没有人。对吧？用不了多久，你的代码库就会变成一团乱麻，谁都看不懂。现在，除了所有的源代码之外，我们还得考虑 token 管理。因为 token 很贵，你们有些人可能知道，对吧？在规模化的场景下，token 是一笔实实在在的成本，我们必须把它算进去。如果你们公司里每个人都开始用 10 倍、甚至 100 倍的 token 怎么办？或者万一你一个月的预算不小心一天就花光了——这事还真发生在我一个朋友身上。对吧？如果你得优先决定这笔开销花在哪里，你知道该花在哪吗？你现在有没有可视化的能力，知道 token 都花到哪儿去了？你们看，就这么简单的几个环节，我们就发现问题了，而且很显然，这里会出现一些很有挑战性的二阶影响。

好，咱们继续聊测试和版本控制。这些有点特别。你们有没有算过测试基础设施要消耗多少算力？我不知道你们怎么样，但在谷歌，我的测试从来就没快过。我从来没说过："哎呀，今天测试跑得比需要的还快了，我的测试算力还有富余。"对吧？每个提交到版本控制的代码都必须跑测试。但另一方面，agents 又特别喜欢跑测试，因为测试能告诉它们干得怎么样。所以，又来了，agents 在做额外的工作，而我手头的工作也更多了。所以，提交量是 10 倍，加上 agents 在那儿折腾，你的测试算力要花多少？实际上，可能会更糟。因为我们在谷歌观察到的是：随着代码库的增长，依赖图的增长是二次方级别的，不是线性的。所以，如果你的代码库变成 10 倍大，而你又想测遍所有依赖关系，确保不会出问题，那你要跑的测试可能要涨到 100 倍。也许是 1000 倍。这将是一个非常有意思的挑战，迟早会出现在你的预算表上。这件事你真的得好好想想。所以，你不仅测试跑得更频繁了，测试体量还可能变成 100 倍。说实话，如果你觉得这都不是事儿，不担心测试算力的开销，那我反而更担心了，因为这说明你一开始的测试就不够多，那些 agents 就是在你的代码库里乱搞，根本没办法知道哪些东西是能用的。对吧？

好，咱们假设你把这些问题都解决了，编译搞定了，测试也搞定了。那咱们聊聊你的版本控制系统。大多数流行的版本控制系统根本没有为性能做过优化。一点都没有。它们优化的是一致性、顺序，对吧？这是它们的本职工作——维护一份完整的记录，而不是跑得飞快。你的版本控制系统性能怎么样？每分钟能处理多少个 commit？我敢保证，比你想的要少。对吧？它根本撑不住你需要的那种 10 倍的提交速度。你上次想版本控制系统的性能是什么时候？想过吗？除非你在 Git 团队干活，否则估计压根没想过。说实话，咱们要是聊到版本控制性能这事儿了，说明问题已经很严重了。对吧？我们这是在挖开发体验的地基了。地基。但这就是系统级变革会发生的事。系统级变革会把你系统的每一个角落都翻一遍，然后说："嘿，你注意到了吗？这儿有个你没料到的事儿。"顺便说一句，你们中那些觉得自己可以用一堆小代码仓库来解决版本控制问题的，我有个消息告诉你。去问问那些用过这策略的人，管着几百个、几千个小仓库的，我跟你保证，那会是一整套全新的麻烦，AI 也未必能帮你省心。对吧？

到目前为止，我们看过的每一个节点都出了问题。而且你信不信，我们看过的还只是相对容易发现的算力节点。这些都是你能直接算出来的东西。就是你拿个数，乘以 10，然后问：这会变好还是变坏。还有很多其他你没预料到的挑战。所以我准备快速过一下我觉得你应该担心的几个点。第一个，验证问题，不光是算力的问题。现在的验证策略可能大概就是一堆单元测试，再加一些集成测试。但代码量涨 10 倍、服务数也涨 10 倍的情况下，集成测试会成为质量策略里最重要的部分。你们有多少人对现在的集成测试方案满意？我跟直播间的朋友说一下，现场一个人都没举手。好，我就知道。我自己也不满意。对吧？集成测试真的很难，我现在也没有趁手的工具能按我想要的方式来做。

然后你还会遇到一个我称之为"布尔组合爆炸"的问题。现在要发布软件，你要求所有测试都得通过。所有布尔值都得是绿的，才能上线。这事儿听起来挺合理。但如果有一天你有一百万个测试，而跑这一百万个测试的底层测试基础设施的可靠性本身就有问题呢？那可能就没办法做到所有布尔值都为真才能发布了。所以你就需要一套新策略。大概得用统计学方法来判断该跑哪些测试，因为我没法全跑一遍。

好，那超大规模的变更呢？我们现在能做重构和大规模改动，这事儿确实很让人兴奋…… 到处都是各种语言和框架。每个人都能做到。但你有相应的流程或社会契约，来让人们处理动辄成千上万行、几十万行、甚至上百万行的合并冲突吗？可能没有吧。所以我们需要弄明白：怎样才能建立一套工作流，去支持超大变更集之间彼此顺利通过？如果公司里每个人都能上手，我们就得拿出新的策略了。

顺便问一句，你见过"智能体编辑战"吗？就是一个智能体做了个修改，另一个智能体进来就说"我不喜欢这个改动，咱们换个做法"。看着挺好玩的——直到你意识到，两边的 token 你都得掏钱。继续往下聊发布。你今天给客户发布多少次？一天一次？如果能做到一天一次，那已经很棒了。如果还没做到，这里有个问题等着你：你会得到 10 倍于现在的代码量。这些代码总得有个去处吧。如果你不是每天都发布，那每次变更的体量估计会变得特别大。而我的朋友们最爱跟我说的一句话就是：超大改动往往特别吓人。这种事咱别干。但代码总得落地，得部署出去才能产生价值。

所以你很可能想尝试更频繁地发布，这是好事。DORA 那帮朋友会很欣慰的，他们会说"对，多发点"。但到一定程度，收益就开始递减了。每秒钟发一次大概也带不来多少价值。介于"每秒发一次"和"还做不到一天发一次"之间，应该有个合适的平衡点。但代码量还是会涨的，我们得琢磨清楚：把这些代码放到哪里，才能不引入更多风险？

再说说内部 API 这块。咱们一直在聊怎么写软件，但公司内部的开发环境呢，那些 API 和数据呢？我最近一直跟身边的朋友说：你那些内部 API，突然之间全都变成了"对外公开"的。它们需要享受到你本来要暴露在公网上的那套加固防护。为啥呢？因为智能体不会跟你讨价还价。它找到一个 API，就开始调；要是能拿到你的数据，它就一定会调。这点我敢打包票。只要一个智能体能访问某个数据集，那数据对它来说就是开放的。所以，如果你对内部数据集和内部 API 没下过同样的功夫，后面就会撞上一些相当棘手的挑战——智能体会发现一些你大概压根不想让它们发现的东西。

再聊聊"杰文斯悖论"。这个词大概都是过去这 12 个月里才听说的吧？之前没人知道这是啥，也不知道怎么念。但现在大家都懂了杰文斯悖论。它的意思就是：一种资源越便宜、越高效，我们用得就越多。在 token 这件事上简直体现得淋漓尽致。我们把它撒得到处都是。它正在改变我们的工作方式，也改变我们对工作方式的思考方式。我们现在在优化——或者说其实没怎么优化——我们其实是把以前那些看不见、隐性的生产力工作，给它们标上了一个明确的成本。那这会怎样改变我们的行为方式？我现在还说不准。

另外，token 引擎放在哪里，你得留个心。要是把承载关键业务的 token 引擎放错了位置，奇奇怪怪的事就会冒出来。举个例子：要是你的回滚流程要靠智能体有足够算力才能跑得动呢？万一有人把那个智能体的 token 预算烧光了，你就回不去了——这大概不是啥好事。

说到回卷，大家知道现在回卷为啥能起作用吗？根本原因就在于：你发布软件的速度，稍微比你发现生产环境问题的速度慢一点。要是你发布软件的速度飞快，比你察觉任何异常的速度都快，那你回滚的姿态会变成什么样？每一次回滚，都得顶着后续一连串互相冲突的变更一起落地。所以，光是发布更快还不够，我们得把整个系统、回滚这件事一起考虑进去。回滚可是一个特别重要的安全阀。

再说说"人人都是构建者"这个理念。这次大会一直在庆祝这个点，我觉得挺酷的。把工程能力民主化，听着确实很酷——直到你发现，你已经把工程给彻底民主化了。你们公司里肯定有那么一个工具，你不喜欢，恨不得赶紧把它换掉。不管是啥，我不在这儿点名哈，但你心里一定有这么一个工具，特别想自己 vibe code 一个出来对吧？把这个想法乘以公司里的每一个人，再乘以他们用的每一个工具。当人人都用着完全不同的工具时，公司的协作纽带会变成什么样？运气好的话，你们还有个统一的数据底座，这是好事，所有的数据都往一个地方流。但如果没这回事呢？"人人都是构建者"是挺酷，直到你得去维护每个人造出来的那堆东西。

也替我们即将让所有初级开发者"加速通关"成为技术管理者这件事捏把汗。为啥当上 tech lead 要那么久？因为你需要积累直觉、判断力和专业经验，才能在拍板的时候心里有底——毕竟带团队的时候，决策的爆炸半径可比自己一个人干活大多了。当一个刚毕业的应届生一上来就面对 50 个智能体随他调遣，可身上既没有直觉，也没有判断力，会出什么事？我怎么在 6 个月里教出别人 10 年积累的经验？说实话，我现在还不知道。
