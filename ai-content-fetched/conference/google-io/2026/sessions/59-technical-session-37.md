首先你需要把 Firebase Logic 集成到项目里，集成之后只要写几行 Kotlin 代码，就能直接调用云端运行的 Gemini 模型。你会先实例化一个生成式模型，然后传入你想用的 Gemini 特定版本。然后基本上只要调用 startChat 就能创建一个聊天会话。这个 SDK 特别酷的一点是它会自动帮你管理聊天历史。你的应用跟模型之间的通信完全由 Firebase Logic SDK 处理。

Google 云上有三个系列的 Gemini AI 模型可以用。有 Gemini Pro，处理复杂任务很强；有 Gemini Flash，延迟特别低；还有 Gemini Flash-Light，智能度和成本的比值很棒，而且延迟也低。

我们想做一个对博物馆非常了解的聊天机器人。前面我提过云端模型存储了大量的世界知识，所以你可以假设 Gemini 对卢浮宫已经了解很多了，但它不一定什么都知道。比如你问它卢浮宫现在的临时展览是什么，聊天机器人基本上会告诉你去官网自己查，并没什么太大帮助。

为了避免这种情况，我们可以用一种叫 grounding（信息锚定）的技术。在模型的上下文窗口里——也就是传给模型的完整输入——我们加入一些相关内容，帮助模型更好地回答用户的问题。Grounding 技术在 RAG（检索增强生成）里也经常用到。你可以自己实现，但我们也提供了几个 API 让这件事变得简单。

有 URL grounding，可以让你使用某个特定 URL 上的文档；有 Maps grounding，用的是 Google 地图的数据；还有 Web grounding，用的是实时网络数据。我们来看看怎么用。

我们先从 URL grounding 开始。假设你想优先使用某个托管在网上的特定页面的数据，那你在实例化模型时把 URL context 工具加到工具列表里，然后在用户提示里告诉模型去查看这个 URL 的页面，基于这个页面的内容来生成回答。这里我们用的是博物馆的 FAQ 网址。现在我们只用了 1 个 URL，但每次请求最多可以用 20 个 URL。

现在如果我们问聊天机器人，比如怎么才能拿到卢浮宫的折扣门票，Gemini 就会真的用 FAQ 页面的内容来回答。比如我说我家有一只叫 Mona 的猫，能不能拿门票折扣，聊天机器人就会回答不行，而且会准确引用 FAQ 里的内容说明光有猫是不够的。

如果我们想让聊天机器人能从 Google 地图获取信息，可以用 Google Maps 工具。这样聊天机器人就能帮你在博物馆附近找咖啡馆或餐厅，比如参观完之后想吃饭。加入这个工具之后，我问卢浮宫附近有哪些适合带小孩去的餐厅，聊天机器人就会给出一个非常全面的回答，它会去查 Google 地图的数据，返回一个餐厅列表，而且这些餐厅都是明确以有儿童菜单著称的。

我们还可以用 Web search grounding，让聊天机器人去网上搜索最新的信息。这样我们就拥有了一个相当给力的博物馆聊天机器人，可以帮我们做参观前的准备。酷的是，我们用 Gemini Flash 和 Firebase Logic 就把这一切都搭建起来了。其实 Gemini Pr……（后面内容未完）

我有一个很合适的用例。评价功能是现在大多数 Android 应用里都很关键的一部分。在你的 app 里，你可能让用户去看别人的评价，或者写一条商家评价，再或者是评价一段内容，比如一首歌、一部剧。你经常会要求用户写评价，有时候还会让用户在 Google Play 上给你的 app 写评价。这对你很重要，但对用户来说有时候挺麻烦的。

我们觉得 Gemini Nano 在这个场景下能帮上大忙。但你也需要让那些还不支持 Gemini Nano 的设备也能用上这个功能。之前在 Jetpacker app 里，chatbot 帮用户推荐餐厅，那如果我们在 app 里加一个餐厅评价功能呢？你可以看到这里有一个话题列表，让用户选择他们对餐厅的感受，然后让模型帮他们起草一段简短的评论，用户在发布之前可以随时编辑。

你可以看到我们选了一些话题，选完之后点击"生成评论"。如果设备支持 Gemini Nano，就由 Nano 来生成；如果不支持，就默认用云端的 Gemini Flash 来生成。对用户来说完全是透明的，对开发者来说实现起来也很简单。

这个 API 支持四种推理模式：prefer on device、prefer cloud、only on device，还有 only cloud。现在已经可以用了，你可以在 Firebase 文档里了解更多然后试一下。

我们刚才讲的其实是一个比较简单的路由逻辑，主要就看两个条件：设备是否支持 Gemini Nano、设备是否联网。但有时候你需要更复杂的逻辑。

比如你可以监控设备的系统状态，如果 CPU 过载或者电量低于某个阈值，就可以切到云端模型。延迟也是一个值得关注的指标，如果延迟超过一定阈值，你就可以切到端侧。还有一点，不是每个请求都需要云端的大模型，有些简单的请求用一个轻量级的 LLM 就行，比如 Gemma 3，只有 2.7 亿参数，可以用来根据请求的复杂度做路由。

我这里只列了三个标准，但实际上还有很多其他维度要考虑，取决于你的 app 需求和你想给用户提供什么样的体验。

后端搭好之后，还有一个关键问题：它跑在云端，我们有安卓应用，那两边怎么通信呢？用什么消息协议？我们可以自己搞一套，自己定义通信模式，决定发送什么请求、返回什么响应。但一旦后端系统的复杂度上来了，自己搞的那套就会变得很难维护、很难处理。

所以我们可以使用 AG-UI 协议。这个协议标准化了智能体和 UI 客户端之间的通信。它预先定义了一组消息类型，专门用来表达智能体和客户端之间的交互。比如智能体可以告诉客户端它的生命周期事件、文本消息、工具调用、状态管理等等。反过来，安卓应用也可以发送文本消息、提供工具调用的结果、发送自定义事件。

现在大多数智能体框架都有 AG-UI 适配器模块，这样它们就能用 AG-UI 协议来交互。客户端这边也有好几个 AG-UI 的库，比如安卓端就有一个 Kotlin SDK，给了你很好用的 Kotlin API 去跟任何支持 AG-UI 的后端通信。

对安卓开发者来说，到这一步你要做的就是：跟你的后端建立连接，然后决定怎么把服务器发过来的 AG-UI 消息可视化出来。在我们的场景里，我们就是把收到的消息解析出来，在好看的界面上展示。

这已经挺不错了，但我相信我们还能做得更好。第二步——决定怎么把传来的事件可视化，这其实要做挺多工作，要琢磨很多东西，可扩展性也不太好，而且还有版本依赖的复杂性，比如后端升级了，但安卓应用还是旧版本，怎么办？

那有没有可能反过来：让应用描述自己支持渲染哪些 UI 组件，然后让 AI 模型直接"说出" UI 呢？这就是 A2UI 协议——它能让智能体在客户端渲染 UI。它和 AG-UI 配合起来特别顺，AG-UI 负责传输。客户端只要告诉智能体自己能渲染哪些组件，然后跑在服务器上的智能体就会决定：用哪些 UI 组件最好、用什么样的层级结构最好，去完成它要做的事，比如问用户问题，最终达成它的目标。

对安卓开发者来说，你唯一要做的就是定义自己支持的组件集合（可能就是你 app 里已经有的那些 composable），提供给服务器，然后把它们渲染出来就行。
