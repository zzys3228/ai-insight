## 📝 内容总结

**《使用 Gemini API 构建智能体》核心要点**

一、核心主题概述

本演讲介绍Gemini Interactions API作为统一接口，用于与模型和智能体交互，涵盖Live API实时对话、托管智能体和自定义智能体创建。

**二、关键发布/技术要点**
- 要点1：Gemini 3.1 Flash Live——多模态实时对话模型，支持90种语言
- 要点2：Live API——有状态WebSocket连接，支持语音到语音处理、工具调用、实时转录
- 要点3：Interactions API——统一接口，可选择与模型或智能体交互
- 要点4：服务端状态管理——自动维护多轮对话上下文
- 要点5：托管智能体——Antigravity预览版，拥有独立沙箱环境执行代码
- 要点6：步骤数据模型——每个动作是独立步骤，不再强塞为"用户轮"和"模型轮"
- 要点7：智能体环境持久化——文件共享、技能加载、环境ID复用

**三、落地应用**
- 案例1：电台点歌机演示——Gemini Live调用Lyria生成音乐
- 案例2：天气仪表板——一次API调用，智能体自动拉取天气并生成HTML
- 案例3：SVG Emoji生成器——通过CLI skill创建自定义智能体

**四、总结**
Interactions API是构建智能体的统一入口，Live API支持实时语音视觉交互，托管智能体提供独立沙箱执行复杂任务。开发者可通过Skills和CLI轻松创建和管理智能体，无需自建基础设施。

我们昨天在开发者主题演讲上发布了一个东西。它基本上给了这个 agent 和 Gemini 一个完全远程的沙箱环境，可以在里面跑代码。待会儿我们会玩一下，自己搭一个 agent。

但 Interactions API 还有一个特别的地方，就是它有**服务端状态管理**。在讲 Interactions API 之前，如果你用过 generate content 或者其他 API 来实现多轮对话，你肯定得自己在客户端维护一个数组，把用户的轮次、模型的轮次全都存进去。这个数组很容易变得特别大，而且你可能不知道该不该把某些轮次给砍掉。

但用 Interactions API 就简单多了。你打第一个请求，比如"嘿，我叫 Will"，返回结果里会给你一个 ID。下一次轮次你问"嘿，我叫什么名字"的时候，模型就能拿到之前所有的上下文。所以你可以理解成，我们在服务端把所有的上下文都帮你堆好了。基本上这就是搭一个多轮应用的全部操作了——你只要传 ID 进去，对话就能继续下去。

另外，Interactions API 还支持**音频生成**功能。如果你喜欢刚才 Thor 演示的语音生成，你可以用 Interactions API 配合我们的 Gemini 3.1 Flash TTS 预览模型来用。API 接口是一样的：定义模型、定义输入、定义响应模态。这里我们想生成音频，所以还要配一下 generation config，因为相比 Puck 的声音我更喜欢 Thor，所以我在 speech config 里配了一下，可以指定声音的名字。然后发起请求就行了，就跟其他模态的调用一样。

所以你看，**不管是什么模态，不管是什么功能，代码片段长得都差不多**。这对我们人类读起来很友好，对 agent 来说也很方便——有统一的模式，我们和 agent 都更容易把这些代码补全。

再来看一个工具调用的例子。搭建一个 agent 的话，有几个关键组件：一个模型（大脑袋），得有工具（相当于手）让模型能跟环境交互，还得有个环境或者循环机制。

用 Interactions API 搭一个纯客户端的简单 agent，你需要定义一个函数或者函数 schema。比如这里有个 file_incident，类型是 function，然后写个 JSON schema，定义一下你期望 agent 传入哪些参数。

接下来在 Interactions API 的调用里，有个 tools 字段。在这个字段里，你**不只能传自定义函数，还能和服务端的工具组合起来用**。Gemini API 自带一些服务端工具，比如 Google Search、Google Maps、URL context，这些你客户端啥都不用做，全在服务端搞定。而且现在你可以把它们和你的自定义函数混着用了。

比如你发一个请求，模型可以自己判断："等等，我得先用 Google Search 搜一下，然后再调用自定义函数。"Google Search 是服务端自动完成的。如果模型决定要调一个函数，它会把这个函数调用返回给你，然后你在客户端自己处理就行。

Interactions API 有一个特别大的转变，让它变得**原生支持 agent**，就是我们重新设计了数据模型。以前所有的 LLM 基本都是为了聊天设计的：用户说一句，模型回一句，用户再说一句，模型再回一句。但 agent 时代这套就不太对劲了——我们有函数调用、环境响应、推理、客户端工具、服务端工具、上下文压缩……这么多不同类型的步骤，根本不能简单归类为"用户轮"和"模型轮"。

所以 Interactions API 改成了**步骤（steps）数据模型**。每一次交互里的每一个动作都是一个独立的步骤。我们有 user_input 步骤，也有 function_call 步骤、function_result 步骤等等。你不再需要硬塞一个 role: user 然后再传函数结果回去——之前这种写法我自己都觉得挺别扭的。现在你有一个专门用来表示函数结果的 action。

看代码片段，这就是一个模型的响应，里面有一个专门的 thought block（包含 summary 和 signature），然后是模型的输出。很重要的一点是，我们把**内容的表示**和**动作/步骤**分开了。比如 thought 的 summary 可以是文本类型，也可以是图片类型。同样，模型既能输出文本，也能输出图片。
