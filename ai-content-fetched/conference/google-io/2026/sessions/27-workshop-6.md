# 会议演讲总结

**标题：创建高级数据驱动的 Gemini API 应用**

本演讲由 Google DeepMind 开发者体验团队的 Mark McDonald 主讲，重点介绍了如何利用 Gemini 的文件搜索 API 快速构建强大的 RAG（检索增强生成）系统。

**核心要点：**

1. **RAG 的价值与挑战**：RAG 是突破大语言模型上下文窗口限制的常用方法，但传统 RAG 系统架构复杂，涉及向量数据库选型、文档分块、检索优化等多个难题，会显著拖慢原型开发。

2. **文件搜索工具的优势**：作为 Gemini API 内置的全托管 RAG 解决方案，它由 Gemini 嵌入模型驱动，支持文本语义理解，并内置自动 OCR 功能，可索引 PDF 等多模态内容而生成纯文本索引，大幅简化了摄入和检索流程。

3. **智能体 RAG 能力**：文件搜索作为工具提供，模型可反复调用、迭代优化查询，实现当前最高效的 RAG 模式之一。例如在请假流程查询场景中，模型能自主多次搜索，逐步定位相关表格与审批流程。

4. **极简开发体验**：开发者只需初始化文件存储并上传文件，无需手动配置分块策略或 PDF 预处理，搜索直接集成于现有的内容生成 API 中。

5. **重要提醒**：尽管流程自动化，仍需人工核查模型输出。

## 📝 完整文字稿

你试过搭建一个完整的 RAG 系统吗？嗯，RAG 确实很难。在这个视频里，我会教你如何用 Gemini 的文件搜索 API 快速搭建强大的 RAG 系统。另外，如果你愿意看到最后，我还会展示一些进阶的数据处理功能。我是 Mark McDonald，是 Google DeepMind 开发者体验团队的一员，主要负责 Gemini 相关的工作。这个视频不需要任何编程经验，但它的目标受众是正在开发 AI 应用的开发者。所以，如果你有一些开发经验，哪怕是写代码的经验，都会对你有帮助。我们开始吧。

如果你开发过应用，你就知道数据很重要。你的应用里有一些对你和你的用户都很重要的内容。也许是你的个人待办事项应用，也许是你搭建的分析引擎。当你用 Gemini API 这样的工具构建 AI 应用时，你会发现 AI 本身并不是什么魔法。真正的魔法来自把你的"独门秘方"（比如你的待办清单）和有针对性的 AI 提示词结合起来。让你的用户能够问出像"我的待办清单里漏了什么？"或者"我忘掉的重要事项在哪个文件里？"这样的问题，这样你就能利用你独特的价值主张来创造神奇的体验。

不过，大语言模型是有限的。它们有上下文窗口，限制了单次请求能处理的数据量。而且就算它们是无限的，token 越多，响应越慢，成本也越高。能用一百万个 token 确实很强大，但有时候少即是多。

RAG，也就是检索增强生成，是突破模型上下文窗口限制的常用方法。这个视频不会深入解释它，但本质上就是：把你所有的内容放进一个数据存储里，然后当你需要回答一个问题时，你会有选择地把相关的片段放进你的提示词里，这样你就只用处理相关的信息。听起来很简单，但事实是 RAG 可能相当复杂。RAG 有很多种形式，从普通的朴素 RAG 到智能体 RAG、自反思 RAG、图谱 RAG、迭代式 RAG 等等。在架构层面上，在你开始设计系统之前，你就已经有很多选择要考虑。然后，RAG 系统的每个组件也需要考虑。用哪个向量数据库？它能跟我的环境兼容吗？我怎么部署它？然后，你还需要考虑如何优化文档处理，比如分块大小和滑动窗口，以及如何处理像 PDF 这样带有表格和其他非结构化数据的复杂文件类型。然后，你的检索步骤也很复杂。怎么把用户的问题或提示词映射到数据存储的搜索上？你要不要设置查询扩展或文档重排序？要考虑的东西太多了。在一个我们追求快速得到答案、AI 承诺快速结果的世界里，这真的会让你的原型开发慢下来。

这就是 Gemini API 的文件搜索工具发挥作用的地方了。文件搜索是 Gemini API 内置的 RAG 全托管解决方案，把大部分索引和检索阶段都封装起来了，这样你就能专注于你的核心逻辑。它由 Gemini 嵌入模型驱动，这是一个基于你所熟知和喜爱的强大语言模型相同架构的、最先进的嵌入模型。这确保了文本的匹配不仅仅是基于相似度，还基于理解。而且它内置了自动 OCR 功能，所以你可以索引像 PDF 这样的多模态内容，但生成的索引完全是文本的。

跟普通的 RAG 流水线类似，这里有两个主要阶段：摄入（提前做好来把你的数据建立索引）和搜索。搜索在这里比传统的 RAG 设置还要简单，因为它只是一个你告诉 Gemini API 的工具。然后，Gemini 模型就会用这个工具来找到相关的文档，把它们加到上下文里，并生成完成提示词所需要的任何输出。

文件搜索作为工具提供的一个强大好处是，模型能够反复调用这个工具，检查结果，并优化或更新它用来找到正确信息的搜索查询。我们把这叫做智能体 RAG，这是目前已知的最有效的 RAG 引擎之一。

在这个例子里，一个用户发了一个查询问怎么请假。在你的应用发起的这一次生成内容调用中，Gemini 会反复用文件搜索工具来调用，回答这个问题。这就是智能体的部分。用户的查询是模糊的，因为可能有不同类型的假期适用。所以，Gemini 不会停下来问用户，而是先发起一个查询来找到所有可能适用的不同类型的请假流程。然后，在读了这些搜索结果之后，它知道用户需要填写特定的表格，所以接下来就搜索这些表格。然后，因为流程需要审批，又做了一个额外的搜索来找出这一步。最后，一旦流程被理解了，并且在模型的上下文窗口中被很好地记录下来，它就能给用户生成一个详细的回答，说明对于他们请求的任何类型的假期，分别需要哪些表格和流程。

正如你所看到的，仅仅通过你的应用的一次生成内容调用，模型就能编排大量的处理过程。所以，仍然重要的是，在使用任何输出之前，都需要让人来检查一下。
