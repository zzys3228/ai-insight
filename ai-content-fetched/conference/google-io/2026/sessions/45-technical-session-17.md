一、核心主题概述

本演讲围绕用户账户生命周期，系统介绍了现代化身份验证的最佳实践和可用API，旨在消除密码风险、降低用户流失、实现抗钓鱼认证。

**二、关键发布/技术要点**
- 要点1：身份联合（Federation）——推荐"先联合后升级"模式，用可信身份提供商一键完成注册
- 要点2：通行密钥（Passkeys）——比密码更安全、更简单、防钓鱼，登录速度翻倍、成功率提升四倍
- 要点3：邮箱验证协议——浏览器内完成邮箱验证，无需收件箱切换
- 要点4：数字凭证API——以隐私方式验证用户属性（如年龄、真实姓名）
- 要点5：Signal API——同步通行密钥状态，确保凭证列表准确性
- 要点6：即时UI模式（Chrome 148）——智能选择最合适凭证，统一展示通行密钥和密码

**三、落地应用**
- 案例1：Pixiv实现99%登录成功率
- 案例2：阿迪达斯使用条件创建功能，通行密钥创建量提升8%

**四、总结**
账户生命周期管理应逐步淘汰弱登录方式，采用通行密钥等现代方案。通过FedCM、邮箱验证协议、数字凭证等API，可在高安全性与无缝用户体验间取得平衡，并为用户提供多设备、多提供方的灵活登录选择。

这是一个完全不用发邮件就能拿到已验证邮箱的网页应用。整个验证流程全都在浏览器里完成。对你的用户来说，这就意味着彻底告别"去收件箱查邮件"这种麻烦的操作。不用切换页面，也不用等邮件送达。

那它底层是怎么工作的呢？这是邮件服务商、浏览器和一个专门的签发机构之间的三方握手协议。邮件服务商可以选择把验证权限委托给这个签发机构。签发机构手里有一个第一方认证 cookie，能证明用户已经登录并且掌控这个邮箱。当用户在你的应用里输入邮箱时，浏览器会向签发机构发送请求来验证这个邮箱。签发机构检查认证 cookie，签发一个安全的数字令牌，然后把它交回给浏览器。浏览器再把这个令牌传给应用。

要想在你的网站上启用这个邮箱验证协议，你需要在邮箱输入框上加上 autocomplete 属性设置为 email，再加一个 N1s 属性。然后添加一个 JavaScript 监听器来监听 email verified 事件。当这个事件触发时，获取这个演示令牌。记得一定要验证令牌的签名和声明内容。

注意，邮箱验证协议这个功能还在开发中，具体实现细节将来可能会根据大家的反馈而变化。它也是一个可选功能，需要各家邮件服务商自己选择是否支持。详情可以看博客文章。

有些情况下，你在最终完成账户创建之前可能还需要验证用户的其他属性，比如真实姓名或者年龄是否符合要求。这有时候是为了符合当地法规。在某些地区，可以使用可验证的数字身份凭证来处理这种情况。你可以用数字凭证 API 以保护隐私的方式来验证用户属性。

想知道怎么用？咱们先从数字凭证三方模型的整体架构说起。首先是**签发方**，就是一个可信的机构，用加密方式给凭证签名，比如可以是发放驾照的机构。然后是**持有方**，就是用户的手机，里面存着这个签名凭证。最后是**验证方**，也就是你的网站或手机应用，通过数字凭证 API 来请求验证。

所以，要想从数字凭证里请求用户属性，你的网站可以调用数字凭证 API。浏览器和操作系统会一起协作，让验证方和用户手机上安装的电子钱包之间能够通信。如果有包含所请求属性的凭证可用，操作系统会提示用户把它分享给验证网站。验证方只收到请求的那些信息，并且直接跟签发方核对真伪。同时，用户那边会清楚地看到自己到底分享了哪些数据。

还记得"先联合登录再升级"这个模式吧？这里用类似的逻辑，等你验证完用户数据之后，就可以提示用户创建一个通行密钥（passkey）用于后续登录。这对应到你网站上的两次 API 调用：先调用 get，用数字凭证 API 验证用户属性；然后调用 create，建议用户创建一个通行密钥。

数字凭证的应用场景已经不局限于政府发放的证件了。数字凭证 API 提供了一个传输层，能把你的应用和用户手机上的应用连起来。它是个多功能的工具，可以分享各种类型的可验证凭证，从证书到登机牌都行。只要用户手机上有兼容的钱包应用，并且里面存着可信签发方发放的凭证，你就可以用数字凭证 API 来验证数据。

>> 谢谢 Natalia。流畅的登录体验非常重要，因为它定下了整个用户体验的基调。登录方式有很多种，这其实是件好事。不同用户可能有不同的偏好，比如密码、第三方联合登录或者通行密钥。我们强烈建议你为用户提供通行密钥支持。通行密钥比密码和一次性验证码（OTP）更简单、更安全，而且能防钓鱼。

首先，我们来看看怎么在一个统一的界面里把通行密钥和其他选项放在一起供用户选择。从 Chrome 148 开始，"即时 UI 模式"可以帮助用户选择马上就能用来登录的凭证。它会展示一个界面，列出用户密码管理器里保存的可用通行密钥或密码。当同一个账号有多个可用凭证时，浏览器会智能地选择最合适的那一个，让用户的凭证管理更简单。跟其他通行密钥请求不同的是，如果没有可用的凭证，浏览器会立刻返回一个错误，不显示任何界面。这时候你的网站就可以把用户引导到普通的登录页面。

需要继续翻译被截断的部分吗？ 好的，我们来聊聊如何创建通行密钥。条件创建、无障碍性……

你应该在新用户注册时或注册后提示他们创建通行密钥。首先，通过调用 getClientCapabilities 并检查 passkeyPlatformAuthenticator 属性，确认用户的设备是否支持通行密钥。确认之后，调用 create 来创建通行密钥。一定要把 authenticatorAttachment 属性设置为 platform，这样浏览器就会优先使用本地可用的凭证提供方。

对于已有用户，我们建议使用条件创建功能，让系统自动创建通行密钥。这个功能可以在不弹出任何提示的情况下触发通行密钥的创建。阿迪达斯（Adidas）就使用了条件创建功能，结果通行密钥的创建量提升了 8%。

要使用条件创建功能，只需要正常发起一个 create 请求，并把 mediation 设置为 conditional 即可。在 Chrome 中，如果用户输入的密码与密码管理器中保存的一致，浏览器就可以在后台静默注册通行密钥，整个过程不会有任何额外提示。

很多平台仅仅通过向用户宣传通行密钥的好处，就看到了转化率的提升。比如 Pixiv 就增加了一个安全确认页面，推荐用户使用通行密钥，因为这种方式登录更快、更安全。就是这么一个简单的页面，就让他们的新用户通行密钥采用率提升了 6%。

另一个好做法是，确保你的代码能正确处理通行密钥创建失败的情况。如果公钥在后端保存失败，可以通过 signal API 向通行密钥提供方发送信号。稍后我们会详细介绍这个 API。

最后，别忘了在通行密钥创建成功后通知用户。一旦通行密钥注册完成，就应该立即告知用户，比如发送邮件或应用内通知，这有助于防止未经授权的凭证添加。

确保用户体验的一致性非常重要，不仅仅是浏览器端，在用户的所有设备上都要保持一致。要想实现跨平台的无缝体验，你可以在你的网站和移动应用之间建立经过验证的关联关系。

对于 Android 应用，你需要在服务器的 assetlinks.json 数字资产链接文件中指定这种关联。你需要填写应用的包名及其 SHA-256 签名证书指纹，并添加 handle_all_urls 和 get_login_creds 关系。别忘了更新应用的 manifest 文件以启用通行密钥。

这个文件用于验证你的网页源（Web origin）和原生应用确实属于同一个主体。没有这个文件，用户就无法用网站上创建的通行密钥登录你的应用，反之亦然。

现在用户已经有通行密钥了。接下来我们要确保你的应用遵循最佳实践，让整个体验尽可能顺畅、毫无压力。

想象一下，如果用户删除了账户，或者在网站上撤销了通行密钥，但通行密钥还留在了他们的通行密钥提供方那里。下次他们尝试登录时，就会看到一个通行密钥，但服务器找不到对应的公钥，登录就会失败。

signal API 可以帮你解决这个问题，它能确保用户的通行密钥保持同步。当后端检测到有凭证 ID 不存在的签名尝试时，就调用 signalUnknownCredential。这会告诉凭证提供方该通行密钥已经没用了，从而在后续登录时隐藏或删除它。

此外，你还应该同步可用通行密钥的状态。每次登录后以及用户删除通行密钥时，都要执行这个操作。使用 signalAllAcceptedCredentials，确保通行密钥列表与密码提供方保持同步。

最后还有一点，别忘了调用 signalCurrentUserDetails，确保通行密钥列表中显示的用户名和显示名保持同步。持续调用这些 signal API 方法，可以保证用户下次登录时看到的提示信息是准确的。

接下来，我们来聊聊打造优秀用户体验的另一个关键要素：你要让用户能够掌控自己的通行密钥。下面是具体做法。

首先，用户应该能够为自己的账户添加多个通行密钥。这一点对于提升韧性非常重要。如果用户能从不同的提供方注册通行密钥，那么当某个提供方不可用时，他们就有备份方案，不必回退到使用弱密码。同时，这也让用户在不同生态系统之间登录变得更加容易。

另一个好做法是，提供一个专门的管理页面，让用户可以查看和管理自己的所有通行密钥。比如，你可以把通行密钥管理和用户常用的密码管理页面或二次身份验证（2FA）管理页面整合在一起。确保用户能轻松找到这个页面，并且页面中有一个集中的通行密钥管理区域。用户应该能够查看、添加或删除通行密钥。

别忘了在做出任何更改时，使用 signal API 通知通行密钥提供方。

在这个管理页面上，你应该显示通行密钥提供方的名称。你可以用 AAID（Authenticator Attestation ID，认证器证明 ID）来查询，比如 Google Password Manager 或 Bitwarden，然后显示对应的 logo。详情可以查看我们的博客。

你还应该清楚地标明通行密钥是否可以同步。如果用户在工作电脑上注册了一个不支持同步的通行密钥，他就无法在家里笔记本上使用这个密钥。你可以通过 create 响应中的 backupEligible 位（简称 BE 位）来判断这一点。

到这里，关于如何打造出色的用户登录体验，我们就讲完了。下一个阶段是用户账户生命周期的下一个环节，我把话筒交还给 Natalia，她会给大家讲讲账户恢复方面的注意事项。

>> 谢谢你，Nina。最后，我们来看看最糟糕的情况——账户恢复。你可能在想："如果用户丢失了设备，或者不小心删除了通行密钥，该怎么办？" 我们为你准备了一些建议，帮助用户重新访问或恢复自己的账户。

我们之前讲过，通行密钥是经过端到端加密并同步的。如果用户…… 好的，我把它翻译成比较口语化的中文：

如果用户丢了手机，但他们的通行密钥保存在像 Google 密码管理器这样的通行密钥提供方里，那他们的通行密钥就会安全地备份到云端。用户可以在新设备上登录自己的 Google 账户，输入安卓设备的锁屏密码，或者 Google 密码管理器的 PIN 码，就能在新设备上访问同步过来的通行密钥了。有了这个通行密钥，他们照样能登录你的应用。很多其他的通行密钥提供方也都支持同步和端到端加密。

要是用户把所有支持通行密钥的设备都弄丢了，你还可以退一步，用经过用户验证的属性来恢复。比如，可以让用户通过当初注册时用的联邦身份账户登录。就算用户一开始没用联邦账户，你也可以用邮箱验证协议，或者我们这次聊过的其他方法，来验证他们的邮箱。有了这些方法，你随时都能帮用户找回应用的使用权限。
