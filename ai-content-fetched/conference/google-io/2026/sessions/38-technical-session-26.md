## 📝 内容总结

## 会议演讲总结：构建智能体基础知识

**演讲人**：Chris Overholt，Google Cloud 开发者关系工程师，主要从事 Agent Development Kit（ADK）相关工作。

**核心观点**：单一 AI 智能体的能力非常有限，仅能回答问题或调用少量工具。一旦涉及协作、支付、流式输出等场景，便需要借助专门的协议来扩展能力。演讲围绕六个关键协议展开，通过搭建"厨房管理智能体"（为餐厅从批发商订购食材）作为实战案例，逐步为该智能体添加协议。

**六个协议要点**：
1. **MCP（模型上下文协议）**：解决工具与数据接入问题。通过 MCP 服务器动态暴露工具，替代硬编码大量工具定义，三行代码即可在 ADK 中接入。
2. **A2A（智能体对智能体协议）**：解决专业能力分散问题。通过标准化的"智能体名片"机制实现跨团队、跨框架的智能体发现与协作，`to_a2a` 一行代码即可暴露服务。
3. **UCP（通用商务协议）**：解决商务交易标准化问题。提供机器可读的商家发现与结构化结账请求，智能体无需解析网页或猜测流程。
4. **后续协议**（音频截断）：涉及支付授权与实时交互式仪表盘的流式输出能力。

**核心价值**：每个协议都针对传统 API 在智能体场景下的具体短板，让智能体能够查真实库存、对接专家、正式下单并完成端到端业务闭环。

## 📝 完整文字稿

[音乐] >> 大家好，欢迎来到今天的分享。我是 Chris Overholt，Google Cloud 的一名开发者关系工程师。我主要从事 Agent Development Kit（智能体开发工具包）方面的工作，也会涉及其不断扩展的工具与集成生态。今天，我想带大家了解六个协议，它们能让你的 AI 智能体在对接外部系统、远程智能体、支付处理以及实时交互前端方面变得更强大。

单凭 AI 智能体本身，它能做的事情其实非常有限。哪怕你给了它一套简单的指令和几个工具，它最多也就是回答问题、调用一两个工具。但一旦你需要让它协作、付钱或者流式输出结果，它就卡住了。如果你和我一样，看到过那一长串智能体协议的缩写，肯定会好奇：怎么会有这么多？它们到底是干什么的？

传统的 API 假设对面坐的是一个人。人会点击"购买"，但智能体不知道如何结账。人能浏览商品橱窗，但 AI 智能体拿不到目录。人能看懂表单，但 AI 智能体只能输出文本。

我们今天要讲的每一个协议，都正好解决上面这其中一个问题。为了让讲解更具体，我们会一起搭建一个"厨房管理智能体"，让它为一家餐厅从批发供应商那里订购食材。我们会使用 Agent Development Kit，也就是 ADK——这是 Google 开源的 AI 智能体构建与部署框架。

先来看一下，如果你给智能体什么协议都不加、什么工具都不给，会发生什么：你让它检查库存并下单，它会信心十足地用瞎编的库存数据、价格和订单确认信息把空缺填上。

为了更深入地理解这些协议，过去几个月我一直在用它们构建智能体，问各种刁钻的问题，琢磨它们到底是怎么解决真实问题的。我们会从一个"光秃秃"的智能体开始，一次给它加一个协议，直到它能查真实库存、下单、授权支付，并且能流式输出交互式仪表盘为止。

到结束时，你就会清楚：每个协议是干什么的、真实代码长什么样、以及什么时候你真正该用它。

首先，从一个你可能已经熟悉的东西开始——MCP，也就是 Model Context Protocol（模型上下文协议）。它帮我们把智能体连到工具和数据上。

第一个问题是：我的智能体需要使用外部工具。没有 MCP 的话，每个 API 端点都得为它写一个定制化的工具。接入几个服务之后，你就已经在维护几十个工具了。这对你和智能体来说，都很难扩展。

MCP 的解法是：让你的智能体连接到一个 MCP 服务器，服务器会动态地对外暴露它可用的工具。这样你就不用硬编码几十个工具定义了，只要指向几个 MCP 服务器，智能体在运行的时候自己去发现能用什么。

对我们这个厨房管理智能体来说，就是连接三个 MCP 服务器：一个对接库存数据库，一个对接菜谱和操作流程，一个对接供应商邮件。在 ADK 里，每个服务器用 MCP toolset 只要三行代码就接好了，你完全不用自己写一个 API 调用。

现在智能体有了真实的数据访问能力，新增数据源也只是再加一个 MCP 服务器的 URL。MCP 最出彩的地方其实是原型阶段——当你还在摸索智能体能做什么的时候，MCP 让你能快速接入新的数据源，看看智能体会怎么用它们，再去决定要不要针对单个工具做优化。

下一个是 A2A，也就是 agent-to-agent 协议（智能体对智能体协议）。MCP 让我们的智能体拿到了数据。下一个问题是：专业能力。厨房管理智能体能查库存，但它并不真的清楚今天的批发价、供应商的品级或者配送窗口。这些知识其实分散在不同的智能体手里，可能是不同团队、用不同框架构建的。

A2A 通过标准化的智能体发现机制来解决这个问题。每个智能体都会在一个众所周知的 URL 上挂一份"智能体名片"（agent card），用来描述它会干什么。你的智能体就去拉取这些名片，看看每个专家能做什么，然后通过 send message（发送消息）跟它们对话。

在 ADK 里，你可以用 to_a2a 这一行代码就把任何智能体暴露成 A2A 服务。在发现侧，你指向远程智能体的 URL，拉取它的名片，然后发送消息。你的智能体不需要知道远程智能体是怎么搭的，甚至不需要知道它跑在什么框架上。

在我们的例子里，厨房管理智能体会发现三个远程智能体：一个管价格，一个管食材品质，一个管物流。所以当你问它："三文鱼什么价？品级怎么样？今天下午 5 点能送到吗？"它就会把每个问题准确地路由到对应的专家那里，完全不关心它们内部是怎么实现的。

A2A 真正有价值的地方体现在组织层面：当公司里每个团队都在造自己的智能体时，A2A 给你提供了一种标准的方式去发现它们、把任务委托给它们。新增一个远程智能体能力，只是再加一个 URL 的事。

接下来，我们的智能体要开始下单了。这就要用到 UCP，也就是 Universal Commerce Protocol（通用商务协议）。它把如何发现商家、如何下单、如何跟踪订单状态都标准化了。

人可以浏览一个店面，但智能体真正需要的是机器可读的商家信息和商品发现能力。UCP 通过 well-known/ucp 这个众所周知的端点提供这些能力。这样，智能体就能发现供应商的目录，并发出一个结构化的结账请求，里面包含订单项、数量、支付明细等。

智能体不用去浏览网站，不用解析 HTML，也不用去猜结账流程是什么样的。代码上就是标准的 HTTP 请求，照着 UCP 规范来就行。这种带类型的请求格式意味着，智能体不用去猜一个结账请求到底长什么样。

对我们的厨房管理智能体来说，这意味着：发现供应商的目录，为 10 吨三文鱼下一个类型化的订单，然后拿到一个确认的订单号。智能体就这样……（音频未完）
