# 使用 ML Kit GenAI 和 LiteRT-LM 在 Android 上部署设备端 AI

## 📋 摘要

**📅 日期**: 2026-05-21 | **🕐 时间**:  PT | **📂 分类**: AI/机器学习 | **🎯 类型**: 技术演讲 | **📊 等级**: Intermediate

**👤 演讲者**: carenc, erintwalsh

通过设备端 AI 让您的应用与众不同，提升用户体验。了解将设备端 AI 引入 Android 应用的不同方式，包括使用 ML Kit GenAI API 实现基于 Gemini Nano 的即装即用解决方案，以及使用 LiteRT-LM 为自有模型定制的使用场景。

## 📝 内容总结

**《使用 ML Kit GenAI 和 LiteRT-LM 在 Android 上部署设备端 AI》核心要点**

本演讲全面介绍了 Android 设备端生成式 AI 的完整技术栈和最佳实践：

**一、端侧 AI 的优势**
- 敏感用户数据可在设备本地处理，无需服务器
- 功能不依赖网络连接，可离线运行
- 无额外推理成本，运行在用户硬件上

**二、端侧 AI 适用场景**
- 处理敏感数据（银行/医疗记录）
- 个性化功能（定制新闻摘要）
- 小上下文窗口场景（短句翻译、内容分类）

**三、技术方案对比**
- **ML Kit GenAI API**：开箱即用方案，自动管理模型更新，提供 Gemini Nano 访问
- **LiteRT-LM**：定制化方案，支持运行小型定制模型

**四、ML Kit GenAI API 新功能**
- AI Core 开发者预览版：可预览最新端侧模型
- 语音识别 API：端侧音频转文本，支持多语言
- Prompt API：支持文字和图片输入
- 前缀缓存：推理速度平均快 2 倍，Pixel 9 上快 4 倍
- 结构化输出：稳定返回一致结果
- 自动提示词优化器（APO）：自动找到最优提示词

**五、LiteRT-LM 定制化流程**
- 选择最小可用模型（如 Gemma 270M）
- 使用 Hugging Face TRL 库微调
- 用 LiteRT-LM CLI 工具转换并量化
- Google AI Edge 门户：100+ 设备基准测试

## 📝 完整文字稿

[音乐] >>大家好，我是Karen，今天我和Aaron一起聊聊Android端侧生成式AI的最新进展。在这次分享中，我们会涵盖以下几个内容：为什么以及如何为你的应用考虑端侧AI用例；AI Core开发者预览版——一种让你可以预览最新端侧模型并改善整体开发流程的新方式；ML Kit生成式AI API——帮助你将大多数用例投入生产，并带来新的能力，如语音识别，以及通过缓存和结构化输出改进的提示API；还有更轻量的TLM，帮助你实现更多定制化用例。

我们先来谈谈为什么应该在你的应用中使用端侧生成式AI。端侧生成式AI指的是直接在设备上处理提示和数据的能力，无需将数据发送到服务器。由于不需要服务器，这带来了一些优势：敏感的用户数据可以在设备本地处理；模型功能不依赖于网络连接，意味着即使网络不稳定或没有网络，模型也能输出可靠的结果；最后，由于一切都运行在用户的硬件上，每次推理都没有额外的成本。

有了这些优势，端侧方案非常适合以下用例：处理敏感的用户数据，如银行或医疗记录；为用户构建量身定制体验的个性化功能，比如根据用户特定兴趣定制的新闻文章摘要；以及需要较小上下文窗口和算力的场景，例如翻译短句或将内容分类到特定类别。构建这些有差异化的用例可以提升用户体验，而Google致力于通过Android上的Gemma和Gemini Nano等模型为你实现这一点。Gemma是一系列先进的开源模型，基于与Gemini模型相同的技术构建。在受支持的Android设备上，Gemini Nano建立在Gemma的架构基础上，并进一步针对电池续航和性能效率进行了优化。这意味着你今天为Gemma 4编写的提示，在下一代Gemini Nano模型随旗舰设备发布时，将自动适配。

为了让你的应用轻松利用这些强大端侧模型的能力，我们提供了一套完整的、生产就绪的解决方案。首先，对于大多数用例，ML Kit生成式AI API提供了最直接的生产路径。这些API通过Android操作系统自动管理模型更新，提供对Gemini Nano的访问，因此应用始终使用Google最强大的端侧模型。另外，如果你的应用面向小众领域，Lite RTLM允许你在用户设备上管理和运行小型定制模型。

我们先深入了解ML Kit生成式AI API，它允许你直接访问Gemini Nano。在受支持的设备上，Gemini Nano通过AI Core（一个Android系统服务）部署和管理。通过AI Core使用Gemini Nano，应用可以获得以下优势：基于设备硬件的优化，实现更低的推理延迟和减少功耗；无需额外管理模型部署，包括Gemini Nano模型的更新；内置的隐私和安全保障。AI Core被设计为隔离每个推理请求，这意味着请求是独立处理的，并且一次只处理来自一个应用，以降低数据暴露给其他使用AI Core的应用的风险。输入数据和生成的输出永远不会存储在设备上的任何位置。

为了让你对集成ML Kit生成式AI API的开发者旅程有更具体的了解，我们来看看从原型设计到用例生产化需要做些什么。使用AI Core开发者预览版，你现在可以测试Gemini Nano的生产版和预览版模型，让你在Gemini Nano正式发布到生产之前就能访问其最新能力。通过这个应用，你可以下载模型，通过测试提示来探索用例，并且可以控制开发者设置，例如绕过配额限制。预览模型目前可以在Google和特定设备厂商的AI Core支持设备上进行测试。

为了进一步改善使用预览模型进行原型设计的开发者体验，我们引入了模型选择API。这个API允许应用在提示API集成中针对预览模型，以测试新模型的完整代码路径。在通过提示API发起模型推理时，你现在可以指定目标模型。通过提供下载和使用预览模型（如Gemini Nano 4）进行原型设计的能力，我们希望你在模型发布到消费设备之前，就能开始优化提示准确性并探索用例。要开始使用，请查看我们官方文档中的分步说明。

接下来，在完成原型设计和优化提示之后，就该在你的应用中直接实现用例了。ML Kit生成式AI API通过高级接口提供开箱即用的质量，让这一切变得简单。对于常见用例，如摘要、校对、改写和图片描述，去年我们引入了无需提示、只需几行代码的API。为了扩展这些受欢迎的用例，我们最近通过ML Kit的语音识别API解锁了端侧音频体验。这个API可以将来自设备麦克风或音频文件的音频内容转录为文本。在受支持的设备上，该API将使用端侧Gemini模型；在所有其他设备上，开发者可以选择使用传统的端侧语音识别模型。这意味着该API兼容所有运行Android 31及以上的设备。使用端侧Gemini模型，ML Kit的语音识别API提供以下优势：为阿拉伯语、印尼语、荷兰语等语言提供更高质量的转录，需要的存储空间更少。传统的设备端语音识别模型需要为用户使用的每种语言单独下载一个100到200兆的模型。但设备端的Gemini模型只需要一个小小的底层适配器，就能转写所有支持的语言。而且还提供了更友好的开发者API，支持音频流式传输。赶快参考我们的官方文档，开始在你的Android应用里实现语音识别API吧。

如果你想在刚才提到的预定义场景之外实现一些独特的用户流程，或者想要对输出有更多控制，我们最近推出了Prompt API。Prompt API允许你向Gemini Nano发送任何自然语言请求，支持输入文字和图片，输出文字。这个API的多功能性让Adobe Photoshop能以多种方式利用它，从头到尾改善用户体验。从创作流程的一开始，应用就能在导入和导出图片时智能地给图片重命名。这个场景就是利用了Prompt API支持图片输入的多模态能力。

编辑素材时，用户经常要在Photoshop里管理很多图层。通过使用Prompt API，应用现在能在图层面板里自动给那些没命名的图层重命名，就像执行一个操作一样。最后，在直接分享到社交媒体的工作流中，应用还能自动生成标题和话题标签。看到各种应用用Prompt API全面拥抱设备端生成式AI，实现翻译、分类、内容灵感等等场景，真的很令人兴奋。

自去年Prompt API首次发布以来，我们做了几项关键改进，帮助你更好地把这些场景落地。我们来聊聊这些新的API，它们能帮你轻松提升性能和质量，把功能交付给用户。

第一个是前缀缓存。前缀缓存通过存储和复用处理提示词中共享重复部分时的中间LLM状态，来减少推理时间。比如，如果一个提示词里有一段很长的前缀在每次请求中都相同，前缀缓存API可以预先处理这部分，从而节省整体推理时间。为了让大家更直观地了解前缀缓存的效率提升，对大多数设备来说，当提示词有1000个token的固定前缀和100个token的动态后缀时，启用前缀缓存后推理速度平均快两倍。在Pixel 9上，更是快了四倍。

启用前缀缓存有两种方式：自动缓存管理和手动缓存管理。用自动缓存管理时，应用只需要定义提示词中共用的部分。而手动缓存管理则让应用对缓存的创建、查询和删除有更多控制。想了解如何启用前缀缓存，包括代码示例和存储方面的注意事项，请查看我们的官方文档。

接下来，为了在请求特定输出格式时提高准确性，并简化开发者的整体工作流程，我们推出了结构化输出API。这个API允许应用定义对象类，作为Prompt API请求的输出返回。比如，假设我们要实现一个场景：从一封简短的邮件里提取信息来创建日历事件。

用普通提示词的方式，我们可能要用英文写一大段提示词，详细说明要从邮件里提取哪些细节，比如事件的标题、日期和地点，还要附带一个我们想要的JSON输出示例。但这种方式容易出错，返回的结果也不稳定。输出可能是个不完整的JSON对象，或者返回的JSON对象里缺少我们要求的某些字段。

有了结构化输出，我们现在可以定义一个日历事件的数据对象，在发起推理请求时使用generate type content。generate type content会接收一个标记为@Generatable的Kotlin数据类，从而更好地理解期望的输出应该是什么样的。在@Generatable里面，你还可以进一步定义guides，指定数据类里每个字段应该如何生成。这种结构化输出的方式能稳定返回一致的结果，而且用Kotlin数据对象让代码更简洁，无缝融入你的Android代码库。

想开始上手的话，请查看官方文档，了解如何在Prompt API中使用结构化输出。

最后，为了消除提示词工程中的猜测工作，我们推出了针对设备端模型的自动提示词优化器（APO）。APO是一个高级工具，面向希望通过精心准备的评估集来最大化输出质量的开发者。它能帮你自动找到最优提示词，把手动调提示词的猜测工作全部省掉。它使用Gemini Pro和Flash这样的服务端模型，结合你提供的数据集和评估指标，提议改进后的提示词，衡量结果的变化，从而为你的具体场景找到最佳提示词。

想试用早期版本的APO，请查看Vertex AI的官方文档。

开始定制方案的第一步，是为你的场景选择最合适的模型。你可能想用Gemma这样的开源模型，或者自带模型。为了在Android上获得最佳性能，请始终使用能可靠解决你场景的最小模型。我们来演示一下用Gemma 270M来微调一个特定任务——这是一个参数为2.7亿的高效模型。假设一个真实的场景：也许你正在为农民开发一个农业科技应用…… 在印度马哈拉施特拉邦的农村。你的后台用英文提供农业见解和天气预警，但你需要把这些信息翻译成马拉地语，这样用户才能真正看懂。你需要模型捕捉到这门语言的实际细微差别，特别是瓦尔哈迪方言、当地的农耕口语和地方俚语。但由于农村地区没有网络，它必须能离线运行。Gemma 270M 可以解决这个需求，但如果针对这个方言进行微调，效果会好得多。首先创建一个专门的数据集，或者从 Hugging Face 上抓取一个公开的英马对话数据集。这能给模型提供数千对配对的句子。你可以直接在 Google Colab 笔记本里用 Hugging Face TRL 库和 SFT 训练器这些社区工具进行微调。加载你的地方数据集，训练模型，直到它能稳定地生成准确的、本地化的翻译。一旦你的模型会说当地方言，就用更轻量的 TLM 命令行工具，把模型转换并量化成针对手机硬件限制优化的格式。这个工具能转换你微调好的模型，直接输出更轻量的 TLM 文件格式。想了解具体怎么转换，可以去看更轻量 TLM CLI 的文档。当你要把自己定制的更轻量 TLM 模型集成到自己的 app 里时，可以用更轻量 TLM API 来管理会话，并在本地直接写推理。首先，你用模型路径和你偏好的硬件后端（比如 CPU、GPU 或 NPU）来初始化引擎，然后在后台加载你的模型。接着你实例化一个对话对象，它会自动帮你管理上下文窗口和聊天记录。最后，为了给用户那种快速打字的效果体验，你可以用 send message 异步接口。它会返回一个列式数据流，让你能把模型生成的 token 直接流式推送到你的 UI 上。把模型跑起来装到你自己的开发者手机上是个大胜利。但是，你怎么保证你的定制模型在整个生态里都能顺畅运行，哪怕是老款设备呢？Google AI Edge 门户可以自动帮你搞定——它会在 Google 实验室里 100 多台真实的安卓物理设备上对你的更轻量 TLM 文件进行基准测试。这样你就能在真正进入生产之前，看到模型在所有这些设备上的表现。虽然今天我们用的是 Gemma 270M 做例子，但需要强调的是，这套基于 Google AI Edge 的流程对任何开源模型都适用。总结一下，当你需要真正定制化的体验时——比如离线的地方方言翻译器——Google AI Edge 和更轻量 TLM 就是你的解决方案。通过把微调好的模型打包成更轻量 TLM 格式，你就能突破新设备的限制，扩展到更广泛、更多样化的安卓设备上。如果你想动手实操，我们有三个很棒的资源给你。想学习怎么训练自己的定制模型，可以去看 Gemma 实战指南。当你准备把模型塞进 app 的时候，Google AI Edge 的文档里什么都有，从 API 参考到性能评估工具应有尽有。如果你想直接上手代码、今天就试试不同的开源模型，可以在 Play 商店下载 AI Edge Gallery app，或者去 GitHub 上 fork 下来。我们希望通过 ML Kit GenAI API（用于大多数开箱即用方案）和更轻量 TLM（用于更定制化的场景）来实现在端侧 GenAI，能让你更有信心。希望这次概览能给你更多底气。我们很期待看到你们把端侧 GenAI 带到安卓 app 里的各种花样。想开始上手的话，可以点开视频下方链接的资源，覆盖了今天讲的所有内容。>> [音乐]
