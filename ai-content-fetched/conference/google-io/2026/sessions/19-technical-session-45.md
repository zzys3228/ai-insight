## 📝 内容总结

**会议演讲总结：针对不断扩展的Android生态系统的适配性开发**

主讲人Diana Wong（产品经理）与Emily Roberts（开发者关系工程师）围绕Android生态系统自适应开发展开分享。

**核心要点：**

**1. 生态机遇与用户价值**
Android不再仅限手机，已拥有超5.8亿台活跃大屏设备。多设备用户消费力显著更高——大屏+手机用户花费是纯手机用户的9倍，折叠屏用户达14倍，是Android上最有价值的用户群体之一。

**2. 自适应开发理念**
"自适应"不等同于单纯适配平板或折叠屏，而是超越设备形态，打造稳健可靠的应用体验，满足用户对所有屏幕的期待。用户不在乎设备形态，只在乎"能用"或"不能用"。

**3. 框架改进与最佳实践**
Android 17针对SDK 37应用将移除大屏退出选项，强制应用响应方向、可调整大小等变化。需注意拉伸UI、变形相机预览、屏幕外控件等问题。

**4. 新设备形态支持**
已正式推出连接显示屏功能，Pixel和三星Galaxy部分设备支持；Android将支持笔记本电脑，ChromeOS与Android体验全面融合，Android App性能提升并集成Gemini能力。

**5. 工具与示例**
推荐使用Jetpack Compose、Navigation 3和Kotlin Multiplatform，案例包括Notability和BandLab等成功实现自适应体验的应用。

## 📝 完整文字稿

欢迎。我是 Diana Wong，一名产品经理，负责跨设备形态的自适应应用开发体验。>> 我是 Emily Roberts，一名开发者。哦，天哪，你们。刚才那个就是计划好的，我安排了人配合。嗯，我是一名开发者关系工程师，主要负责 ChromeOS 和自适应应用。在今天的演讲中以及整个 IO 大会上，你们会非常频繁地听到"自适应"这个词。自适应并不是只针对平板、折叠屏或者某个特定设备形态来开发，它远不止于此。它是关于超越设备形态的思考，打造稳健可靠的应用体验，满足用户对所有屏幕的期待。如果我给我妈一个平板，她不会去想为什么这个应用会拉伸到屏幕外面，为什么没法拍照。她就想用这个设备做点好玩的事。>> 现实情况是，我们的用户并不会只用一个设备。他们可能在一台折叠屏上开始一个任务，在桌面上的平板上继续，然后在笔记本上完成，期望这种体验能在每个界面上无缝衔接。像 Emily 妈妈这样的用户通常不会考虑设备形态。他们只会想"这个应用能用"或"这个应用不能用"。我们一直在努力改进框架和库，帮助你们的应用在整个生态系统中运行得更好。我们也希望你们能采纳这种自适应的思维方式。>> 所有这些设备形态给你们带来了很多令人兴奋的机会，也带来了一些新的考量。所以今天我们要聊聊 Android 生态系统，它如何随着一些令人兴奋的新型设备形态而扩展，还有 Jetpack Compose 的新 API 和更新。我们会看看包括自适应 AI 技能在内的一些工具，最后我们会介绍几个 UI 框架，你们可以用它们把现有应用带到 Android 上。我们还会穿插一些数量惊人的法式甜点，可能还会有卷饼。好了，咱们开始吧。会很精彩的。

我们先简单看一下生态系统的概况和它的发展方向。Android 设备的世界已经不再只是手机的故事了。用户正在购买使用多个 Android 设备的生态系统，他们期望你们的应用能跟着他们走，不掉链子。现在有超过 5.8 亿台活跃的大屏设备受益于自适应应用的投入，随着我们扩展应用可以运行的界面，这个数字还在不断增长。这不仅是潜在用户数量的问题，我们还观察到拥有多设备的用户花费更多。像平板或者笔记本加手机这种大屏加手机的用户，花费是只用手机用户的 9 倍。对于折叠屏用户来说，这个倍数跳到了 14 倍，使它成为 Android 上最有价值的用户群体之一。Play 商店正在帮助增加安装和消费的机会，让用户知道哪些应用在多设备上表现最好。针对大屏优化的应用现在会在详情页上显示"已为大屏优化"的徽章。从去年开始，越来越多的应用开始走向自适应。这里有一些很棒的应用示例，它们在 Android 上提供了优质体验。

我会深入讲几个，比如 Notability。它的 Android 首秀为优质生产力应用树立了新标准。它使用 Jetpack Compose、Navigation 3 和 Kotlin Multiplatform 构建，能在各种设备上提供直观的自适应体验。它还通过低延迟 Ink API 提供了出色的手写笔支持。另一个例子是 BandLab，一个功能齐全的协作式数字音频工作站。用户可以在多轨环境中创作数字音乐和录制现场表演，当设备有触控板和屏幕空间时，用户希望能够利用这些优势。BandLab 利用 Compose 重写了他们的竖屏社交信息流，现在它可以扩展成瀑布流式的多列布局，让应用在宽屏设备上感觉恰到好处。

自适应开发的好处在于，你做的每一个优化都会自然地提升跨设备的体验。去年，我们为所有应用引入了更改，让它们默认采用自适应行为，移除了设备上的黑边。我们提供了一个开发者退出选项（在目标 SDK 后面），给应用调整的时间，现在在 Android 17 中，当应用针对 SDK 37 时，这个退出选项将被移除。在最小宽度超过 600 dp 的大屏设备上，比如平板、笔记本以及折叠屏和 XR 的内屏，Android 将不再响应某些移动端的限制，更加接近响应式 UI 开发。这包括应用对设置方向、可调整大小以及宽高比的响应方式的变化。请注意，这些限制在手机等小屏幕以及游戏上仍然会生效。我们还在继续推进。你可以让你的应用自然适应在任何窗口中都看起来很棒，而不仅仅是 16:9 的手机屏幕。要注意的问题有：拉伸的 UI、被压扁或歪斜的相机预览、跑到屏幕外的控件或动画，以及旋转、折叠或调整大小事件后应用的状态。请务必用兼容的模拟器测试你的应用，看看这些更改下它会如何呈现给用户。如果你没有 Android 17 设备，你仍然可以在现有的旧 SDK 上测试，进入开发者选项，选择应用兼容性选项。在 SDK 36 下面，启用"默认通用可调整大小"标志。这些 Android 更改有助于为不断发展的 Android 生态系统铺平道路。

让我们看看 Android 各设备形态的新动态。正如我们今年早些时候宣布的，连接显示屏功能已通过最近的 Android 功能更新正式推出。这意味着任何兼容的 Android 手机或平板都可以通过连接外部显示屏，瞬间变成强大的生产力中心。Pixel 和三星 Galaxy 的部分手机已经支持此功能。它让兼容的手机和平板用户进入了桌面级的世界，带来更高的参与度。（待续） 不用额外设备就能做到。能适配鼠标和键盘输入的 App 会脱颖而出。桌面端还不止于此。如果你上周看了 Android 的发布会，你就知道 Android 将会支持最强大、最流行的计算设备形态之一——笔记本电脑。

>> Android App 在 Chromebook 上作为一等公民已经存在十年了。当然，为 Android 做自适应开发，也就是说做好大屏布局和顶级的输入支持，仍然是让你的 App 在笔记本上看起来很棒的最好方式。在硬件方面呢，我们一直在不断迭代，比如更强大的 Chromebook Plus 设备。所以，你能想象我站在这里谈论基于 Android 技术栈打造的下一代 Chrome OS——也就是 Google Book（Google 笔电）有多激动吗！我知道它已经官宣过了，但我还是超级兴奋！新的名字，加上我们即将推出的来自各个 OEM 厂商的出色设备阵容，代表着计算领域的转变。下一代产品将带来强大的笔记本体验，运行数十亿移动用户每天依赖的同一套 Android App。这是一个里程碑，我们应该庆祝一下。但我也认为有必要提一下，这是一个持续的演进过程。我们把多年来在 Chrome OS 上的创新和积累的经验，带到了 Chrome OS 和 Android 用户体验的全面提升中，不仅仅是笔记本上。这些变化有些已经在手机和平板上可见了。比如，平板上的 Chrome 浏览器已经有了书签栏，就像这里展示的一样，手机和平板也支持固定标签页和外接显示器支持。而且这些改进是双向的。下一代 Chromebook 或者说 Chrome OS 也会有提升。你会马上注意到 Android App 的性能提升。它还把 Gemini 的能力带进了你的笔记本工作流，解锁了全新的桌面用户旅程。

我真希望能再多讲一些关于 Google Book 以及所有那些超棒的设备和技术细节，但最重要的是，你的 App 已经准备好了。开始测试的方法之一是使用新的桌面模拟器，它现在已经在 Android Studio 的 Canary Quail Canary 4（我想是这个版本）里可用了，你可以在虚拟设备管理器里的桌面预览里找到它。确保 SDK 是 37，就能体验到窗口缩放的效果了。

在文档方面，为了帮大家准备 Google Book，我们在 developer.android.com 和 material.io 上整理了新的指南，包括针对桌面的设计套件，关于右键和悬停状态这些我超爱的输入方式的新指南，还有更具指导性的布局建议。我特别想分享的一个文档更新是：当你的 Android App 在这些带指针输入的大屏上运行时，有时候信息密度能让 App 在桌面上感觉更对。所以，在大屏上配合精确的指针，UI 稍微紧凑一点反而观感更好。这种差别可能很微妙，比如这个 mockup 里展示的间距和按钮大小。

最后，就像 Diana 提到的，随着这些新 Android 设备的到来，用户很可能不止一台。Continue on（接着用）是 Android 17 里 Activity 类中的一个新 API，它让用户可以在第二台设备上从第一台设备停下的地方继续。你看这台手机上，Google Docs 里我们已经在一个文档标签里翻到很靠下的位置了。所以想在平板上继续编辑的话，要找到那个位置会挺烦人的。Continue on 允许你把数据传过去，用户就能直接在原来的文档标签里、原来的位置继续操作。来吧，GIF……

要在 Android 17 里集成并确保你的 App 准备好的话，只要在清单里把 handoff enabled 设为 true，然后处理启动 Intent 里的那个 bundle 就行了。基本上就这么多。想知道更多关于 Continue on 和这些桌面更新的内容，可以看看这个专门的录播 talk。

>> 再说另一类能受益于自适应开发的设备，车载 App 体验就是这些变化如何应用在车机屏幕上的完美例子。无论是追内容还是保持生产力，用户在车停下来的时候就能享受自己最爱的移动端体验。如果你的 App 已经做了适配，那就很棒了，你已经差不多搞定了。

对于 Android Auto（也就是你的手机在跑 App，但界面投射到车机屏幕上），你可以加一个合适的清单项，然后在 Android Auto 文档里看看具体的资格要求。对于 Automotive OS（也就是你的 App 直接装在车机嵌入式信息娱乐系统里），你的 UI 跟平板一样轻松放大就行。除了车载相关的清单标志之外，还有几件事要确认一下你的 App 能不能装。

首先，在你的清单里检查这些 feature。当然，你的 App 在支持这些特性的设备上还是可以用这些特性的，但如果你的 App 并不需要它们才能跑，你可以用 required="false" 来声明。其次，检查一下你对 Google 服务的依赖。你在手机上用的很多 Google Play 库在车机上也有，但并不是全部，你可以查文档看看是不是不小心把某个库加进去了。
