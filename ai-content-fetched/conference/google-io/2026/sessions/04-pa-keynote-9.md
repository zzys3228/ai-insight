## Chrome 新功能演讲摘要

**演讲主题：Chrome 开发者新动态**

**主讲人：** Paul（Chrome 开发者关系团队负责人）

### 核心要点：

1. **两大新 API 演示**
   - **HTML in Canvas**：实现 3D 沉浸式网页体验，支持 DOM 访问、文本选择等网页操作，并集成 WebGPU 等现代能力。
   - **WebMCP**：允许网站向智能体（Agent）暴露工具，使其能完成多步骤任务（如关灯、重启电脑等）。

2. **行业转型趋势**
   - 演讲者认为，**智能体驱动的网络体验**是自网络诞生以来最重大的计算变革，其转型速度将远超从桌面端到移动端的迁移。
   - 网页正从"用户浏览+操作"的媒介，转向"用户委托智能体完成任务"的新形态。

3. **三大战略方向**
   - **网站侧**：参与智能体生态，确保网络成为一流体验。
   - **工具侧**：让开发工具理解现代网络能力，支持现代基线开发。
   - **未来规划**：公布 Chrome 正在推进的实验性功能，征求开发者反馈。

4. **关键启示**
   - 智能体能通过截图和 DOM 分析网站，但需开发者回归**语义化 HTML、可访问性等基本原则**。
   - 用户意图已从单一交互转变为跨网站、多阶段的任务流程，需要整合性方案来升级网站。

5. **Chrome 内置 AI 集成**
   - 今年早些时候在 Chrome 中推出 Gemini 功能，提供标签页内的浏览器助手，可浏览、总结内容并协助完成任务。

**总结：** 演讲聚焦 AI 智能体时代的网页开发变革，强调 Chrome 正通过新 API 和 AI 能力助力开发者拥抱下一代网络体验。

显然，浏览器本身也在不断进化。今年早些时候，我们在 Chrome 里推出了 Gemini 功能，用户不用切换标签页就能聊天、多任务处理。我们打造了一个浏览器助手，它能浏览网站、总结内容，还能帮你出小测验、做书啊，帮你找到合适的睡袋之类的。它之所以能做到这些，是因为网页本身就描述了上面有什么功能、网站是干嘛的。所以我们的工具就能理解这些。这就回到我刚才说的那个点，如果我们能明确地告诉这些工具——这个网站设计得不错、结构很清晰——那就能帮助智能体和工具更好地完成用户的目标。

如果我们再往前走一步，就像电子表格让每个人都成了程序员一样，我觉得像 Chrome 里的"技能"功能也是这样，让每个人都有能力像开发者一样去指挥浏览器。说到 Chrome 的技能，你可以浏览一个现成技能的库，让它帮你提取产品规格什么的。我听说现在大家都讲究蛋白质宏量营养素，说实话我也不太懂那是什么，但你可以从网页里把这些数据提取出来。你还可以把长文档总结一下，快速找到你需要的内容。然后你可以把那些对话变成"技能"，这些技能是可以重复使用、可以混搭的，而且能在很多不同的网站上运行。我觉得这是一种全新的体验层级，我们要让人们更像开发者，只不过是在一个更高的抽象层面上。

不过话说回来，好的一面是，也是我要讲的那些更偏开发者层面的东西：大语言模型其实可以通过抓取图片、抓取 DOM、猜测点击位置来推断一个页面的意图。但这背后的过程，说实话，有点难懂，也不太透明。而作为开发者，我们其实想要更多的控制权。我们希望能够引导这些工具在我们的网站上按照我们想要的方式运行。这就是 Web MCP 发挥作用的地方了。

Web MCP 目前还是一个拟议中的规范，它的目标是把 HTML 表单和 JavaScript 函数转化成你的智能体可以使用的工具。这样一来，你就能更容易地参与到这种智能体生态系统中来。

关于 Web MCP，我其实有个问题想问大家。你们都知道 Web MCP 是什么吗？看你们的表情……我就知道你们不太清楚。（笑声）

Web MCP 呢，我们的目标是整合 MCP（模型上下文协议）里很多成熟的东西。但我们目前只关注 MCP 里的"工具"这一块。而且你不用专门搭一个 MCP 服务器或者命令行工具，再费劲去跟你的网站做集成，你只需要把当前的浏览器标签页暴露出来，它就能列出一组工具，智能体就能在你的网站上调用这些工具。

就像我之前演示的那个例子，我暴露了灯开关、机器、浏览器本身，这样就能切换页面，这些都是我暴露给智能体的工具类型。当然了，那只是个有意思的演示。想象一下更实际的场景：你开了一家网店，用户想买双鞋，比如十码的棕色鞋之类的。在这种情况下，有了 Web MCP，你就可以暴露一系列工具，比如搜索、加购物车、可能还有结算功能。具体看你想让用户在哪个环节参与。你可以把这些通过 HTML 表单或者 JavaScript 的方式暴露出来。然后基本上，智能体应该能自己判断该用哪些工具、按什么顺序执行，最后顺利完成用户的任务。

这个方案很棒的一点是，我们可以直接用现代 Web 开发里已有的那些语义。大家基本上都知道怎么写一个 HTML 表单对吧？我们只需要在表单上加一些额外的属性，比如一个工具描述，告诉智能体这个表单是干嘛的、怎么跟它交互；再加一些工具参数的描述，告诉智能体应该怎么填这个表单。

比如说，如果你有个星级评分，你就可以标注一下"这是星级评分"；或者有描述字段、评论字段什么的。通过普通的 HTML 表单加上属性，你就能跟智能体解释清楚它该怎么操作。我觉得这真的太棒了。

如果你需要命令式的 API，我们也有 navigator.modelcontext.registertool，用法差不多。你给它起个名字，让智能体知道该调用哪个工具；给它一个描述，说明这个工具的用途；再给它一个输入 schema，描述一下传给函数的数据结构。当智能体觉得这个工具能匹配用户的需求时，它就会带着数据去执行这个函数。我觉得这就是一个非常强大的基础元件，我们既能用 HTML 的方式，也能用 JavaScript 的方式参与到这种智能体生态中。

我最欣赏这个方案的一点是……其实我觉得它……我不想说"简单"，但应该算是相对……好吧，我就直说了，算是相对简单的。抱歉哈。（笑声）

从技术层面来讲，要开始把它集成到你的体验中其实不难。但从更大的层面来看，可能涉及到你整个网站的架构，那就会复杂一点。不过这个方案真正强大的地方在于，因为它运行在浏览器里面，也就是运行在用户的实际环境里。所以像登录状态、Cookie、本地存储这些，工具天然就能用上。这是其他基于 MCP 的工具系统没法比的——那些系统并不是真的在浏览器里运行的。

当然，所有这些都还是在浏览器的安全边界和安全上下文中运行的。但我觉得这是一个很好的开始，能够让我们融入到整个生态系统里面去。

我想给大家快速演示一下我们一个合作伙伴的案例。现在这还只是个实验，不是正式上线的产品，但 Expedia 一直在尝试把 Web MCP 集成到他们的一个假设性的流程里。我就带大家过一遍这个演示。

接下来我还想给你看一个东西，我一直想说的，就是希望让 HTML 成为新的 JSON，为此我们推出了一个新的 API 来帮大家实现。如果你用过框架，那你应该知道一种叫做"孤岛架构"的东西。我觉得这是一个非常漂亮的设计模式和概念，但对我们这些纯写 JavaScript 的开发者来说，要做类似的事情本来就很麻烦。不过现在，新推出的动态标记流式 API 提供了一种全新的改进体验的方式，它通过 streamHTML 和 streamHTMLUnsafe 方法让你可以把 HTML 直接注入到文档里。其实这背后还有一整套相关的 API 接口。

这意味着什么呢？你可以直接把 fetch 请求拿到的 body 流式塞进 DOM 里，再也不用像以前那样先转成 JSON 之类的格式了。我觉得这是一个非常强大、非常新的原语。这些 API 从 Chrome 148 开始就已经可以用了，算是比较新的特性。它们会改变我们构建 Web 体验的方式，至少我个人是这么认为的。

不过大家还是去这个 URL 自己试一下，给我们反馈，这样我们才能把整个平台都铺开。过去这一年，我们其实一直很专注一件事，就是让 Web 感觉更好用，给大家提供更多的 API。你从 API 的角度看就会发现，大量的 CSS 和 JavaScript API 涌现出来，让你能构建更复杂、更丰富的用户界面。但其实这不仅仅是 API 和美学层面的事情，我们的目标是帮你打造那种让人滑起来更顺畅、视图切换更自然、并且让用户能更直观、更接近真实物理交互的网站。

不过有一个技术，我觉得会真正改变我们在 Web 上构建的东西，那就是 HTML and Canvas。我花一两分钟讲讲这个。一开始那个演示里你们也看到了，那种 3D 体验，还有把一个假的网页浏览器集成到电脑里，这些都是 HTML and Canvas 干的。它做的事情就是把标准 DOM 直接渲染到 Canvas 元素里面。用法还挺多的，比如你可以想象那些 2D 应用框架，像 QT 之类的，甚至 3D 引擎，都可以集成这个能力，这样它们就能享受到 Web 那种丰富的语义化 UI 层带来的好处，而且里面跑的是真实的 DOM 内容，能搜索、能翻译、能无障碍访问。

我先给大家剧透一下这个 API 能干的一些事情吧。最基础的一层，你可以拿单个元素来玩。比如这是一个 input type equals range 的输入框，你可以把它做得跟原本的样子完全不一样。从语义上讲它还是一个正确的 input 元素，但我们在外面给它套了一层更丰富的 UI 壳。我倒不觉得每个网站都得做成那样，说实话那也太夸张了。但如果你有自己统一的设计语言，目前在普通网页里实在实现不了，那就可以考虑用 HTML and canvas 来增强一下。而且万一 API 还不支持也没关系，大不了就 fallback 到现在的样子呗。

再往上一层，你还可以把它集成到 3.js 这类库里面，做出那种 3D 风格的体验。比如 3D 的汽车仪表盘、可交互的图书等等。外头有一个专门的演示区，你可以去看看，都有什么玩法。而且所有这些内容全都是可访问的、可交互的，能用上浏览器的所有功能，比如翻译、搜索之类的。我觉得这是我们在 Web 之上又新开的一层表现力。
