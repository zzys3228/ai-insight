# 在你的 AI 编码工作流中解锁现代 Web 功能

## 📋 演讲摘要

**📅 日期**: 2026-05-19 | **🕐 时间**: 16:30 - 17:15 PT | **📂 分类**: AI/机器学习 | **🎯 类型**: 技术演讲 | **📊 等级**: Intermediate

**👤 演讲者**: philipwalton

介绍Chrome团队推出的"现代Web开发指南"，解决AI编码助手在前端Web开发中的三大痛点（过时模式、知识盲区、保守建议），提供基于技能的指南帮助AI智能体正确使用最新Web特性。

---
## 📝 内容总结

**会议演讲总结：《在你的 AI 编码工作流中解锁现代 Web 功能》**

演讲者 Philip Walton 指出，当前 AI 编码助手在前端 Web 开发中存在三大痛点：一是倾向于推荐过时模式（如默认生成 172KB 的 JavaScript 方案，而现代方案仅需 3KB 且无 JS）；二是对最新 Web 平台特性存在知识盲区或掌握错误信息（如 `interesttarget` API 名称已更新但模型仍在使用旧名）；三是浏览器兼容性建议过于保守，未能利用 Baseline 等现代工具。

为此，Chrome 团队推出"现代 Web 开发指南"（Modern Web Guidance），一套由 Chrome 团队及外部专家共同编写、覆盖性能、可访问性、安全性等高阶实践及最新 Web 底层特性的技能包。其工作原理是：在 AI 生成回复前对本地指南进行语义搜索，匹配后将相关指南注入上下文。该工具基于"技能"机制，兼容任何 AI 编程平台，一行命令即可安装。Chrome 团队承诺将持续投入，确保指南内容质量与时效性。

**核心要点：现代Web开发指南解决AI编码痛点、基于技能机制、兼容任何AI编程平台、承诺持续投入更新**

[音乐] >> 嘿，大家好。我叫Philip Walton。今天我想先给大家看一段我录制的两个不同demo的屏幕录像。这两个demo看起来和用起来完全一样——都有平滑动画展开和收起的下拉菜单，都完全支持键盘操作，都支持点击外部自动关闭，并且在Chrome、Firefox和Safari上都能正常运行。这两个demo也都是用AI生成的。实际上，它们用的是同一个编码智能体、同一个模型、同一条提示词。区别在于，左边这个是智能体用标准的、默认的、开箱即用的配置生成的；右边这个用的是同一个智能体，但额外安装了一个东西——这个我等会儿再说。

不过这两个demo之间还有一个很大的区别。如果我打开开发者工具，聚焦到网络面板这里，你可以看到左边这个加载了172KB，其中大部分是JavaScript；而右边这个加载不到3KB，而且完全没用JavaScript。需要说明一下，这172KB是线上传输的、压缩并且开启所有生产优化后的体积。我一行代码都没改过，这就是AI给出的方案。

当然，坦白说，这个例子有点刻意为之，因为没有哪个应用就只有一个设置菜单、别的什么都没有。一旦你加更多功能，可能确实需要多写一些JavaScript。但我确实觉得这抛出了一个很有趣的问题：既然同样的功能可以用完全不用JavaScript的方式实现，而且还能跨浏览器运行，那为什么AI编码智能体偏偏要推荐一个需要将近172KB JavaScript的方案呢？

所以，如果你们今天早些时候参加了其他的分享会，你们已经听了很多关于AI的内容，很多关于AI有多惊人的。说实话，AI确实很惊人，真的，我每天都觉得很不可思议。但是，这只是故事的一面。事实是，虽然AI已经改变了软件行业，让我们开发者的写代码速度比以往任何时候都快得多，但故事还有另一面。AI肯定不是完美的。如果你们跟我一样，在社交媒体上关注很多知名的前端开发者，你们就会发现，这些人非常乐于指出这些AI编码智能体到底有多不完美，尤其是在写HTML和CSS的时候。

你要知道，这些人说的一点都没错。过去几年网页领域出现了非常多令人惊艳的新进展，然而，如果你让一个编码智能体去帮你构建一个网页，它很可能不会去使用最新的Web平台特性来做这件事——哪怕这些特性真的是最合适的工具。这其实也不难理解，因为这些模型用的大部分训练数据，都是在大多数开发者还得兼容老旧浏览器（比如IE）的时候写的。

但不管原因是什么，现实就是：对于前端Web开发来说，编码智能体确实倾向于推荐过时的方案，即使有更好、更现代的替代方案。

我们看到的第二个大问题是，AI模型存在知识盲区。每个模型都有一个知识截止日期，在那之后Web领域出现的任何新进展，模型的训练数据里都没有。就拿过去这一年来说，Chrome就上线了50多项全新的Web特性，但由于这些知识截止日期的存在，这些特性中的很多模型根本都不知道。虽然像RAG和联网搜索这些技术可以缓解这个问题，但并不能彻底解决。

比如说，如果你让一个编码智能体去用它从未听说过的特性，它一般会去搜一下。但如果你让它去做一件它自认为已经会的事情，它通常会立刻非常自信地给你答案，而不去查更新的信息。

举个例子，下面这段代码是智能体在我让它用新的interest-invokers API构建一个工具提示时生成的。仔细看高亮的那一行，你会发现模型用了`interesttarget`这个属性，这是不对的。`interesttarget`是这个API在2023年最初提出时的名字，但在去年秋天这个API正式进入Chrome之前不久，名字被改成了`interestfor`。所以，你屏幕上看到的这段代码在浏览器里运行的时候会静默失败，这调试起来会非常痛苦。

具体到这个API的问题，可能下一代模型就会修好，但更大的问题短期内不会消失。

我们看到的第三个问题是，AI模型是如何做跨浏览器支持的决策的。在我们的测试中，AI在跨浏览器支持方面保守得完全没必要。这些模型或者工具厂商想要优化更广泛的兼容性，这个可以理解，但现在我们有了像Baseline这样的项目，它几乎包含了每一个Web特性的兼容性数据，所以真的没有任何理由不用它。智能体不应该给出那种一刀切、泛泛的浏览器支持建议，而是应该根据你项目的具体需求来给出推荐。

好，这就是我们看到的AI编码智能体在现代Web开发方面存在的主要问题的快速概览。简单总结一下：第一，AI编码智能体经常推荐过时的模式，尽管有更好、更现代的替代方案；第二，AI模型存在盲区，对最新的Web平台特性要么一无所知，要么掌握的信息是错的；第三，AI模型倾向于给出过时的、一刀切的浏览器支持建议。

但好消息是，在Chrome团队，我们相信这些问题都是可以解决的。我们一直在努力开发一个解决方案，我们很兴奋地想跟大家分享—— 好的，我今天给大家带来一个早期预览产品。我们叫它"现代 Web 开发指南"。这个指南是一套全面且经过专家审核的技能合集，覆盖了 Chrome 在构建现代 Web 应用方面的所有最佳实践。这个工具包把 Chrome 团队几十年积累的 Web 平台经验和专业知识，直接嵌入到你的编程 AI 助手里面。而且因为它是基于"技能（skills）"的，所以能用在任何 AI 编程平台上。你只需要一行命令就能安装：npx modern-web-guidance install。或者，如果你正在使用屏幕上展示的这些 IDE 或编程 AI 助手，你也可以直接在这些工具里找到并安装现代 Web 开发指南。

我知道现在市面上新出的"技能包"简直多如牛毛，作为开发者想跟上都很难，甚至有点让人崩溃。更难的是，你根本分不清哪些技能是真材实料值得用的，哪些说白了就是一堆垃圾内容。我想强调的是，现代 Web 开发指南绝不是什么普通的技能包。这是 Chrome 团队过去六个月一直在大力投入的项目，而且我们承诺会长期持续投入下去。

好，我们深入看看里面具体有什么、是怎么工作的。

现代 Web 开发指南包含两层不同维度的指导内容。第一层是一套**高阶指南**，把性能、可访问性、安全性、用户体验等大方向上的 Web 最佳实践都整理归纳好了。除了这些高阶指南之外，我们还有一套**非常具体的底层指南**，专门讲那些最近才加入 Web 平台的新功能。我们发现 AI 模型目前还不太会用好这些新东西。

所有这些指南都是由 Chrome 团队成员编写的，还邀请了一批外部领域专家一起参与共建。因为这些指南聚焦的是 Web 平台的核心功能，所以不管你用的是什么框架、什么技术栈，都能用得上。

在我深入讲这些指南的具体内容之前，先给大家快速过一下它整体的工作原理。

当你直接用一款编程 AI 助手的时候，流程其实很简单：用户输入提示词，提示词传给 AI 模型，模型生成回复。但如果你装了现代 Web 开发指南，那在 AI 生成回复之前，它会先在本地的指南包里做一次语义搜索，看看有没有哪条指南跟你提示词里的需求场景匹配上。如果匹配到了，这些指南就会被加进上下文窗口里。如果没有匹配上，AI 就按正常流程回复。

这里要注意一点，它和大多数技能包的工作方式不太一样——别的技能包每个技能都是一个独立的 skill.md 文件，而现代 Web 开发指南只暴露一个入口技能，通过它去调用一个命令行工具来做搜索。我们这么做是因为现在已经有 100 多条指南了，未来还会再增加几百条。如果每个都做成独立的顶层技能文件，根本扩展不下去。

好，我们再具体看看这些指南长什么样。

刚才我提过，我们既有高阶的学科类指南，也有底层的功能类指南。高阶指南和你之前见过的技能包差不多，但底层功能指南我们换了一种思路，我觉得还挺有新意的，想跟大家说说。

第一点，我们不是去讲功能本身、不是去讲这个功能怎么用，而是每条指南都聚焦在一个具体的**应用场景**上——这个功能能解锁什么、或者让什么事情变得更容易实现。举个例子，我们为 View Transitions API（视图过渡 API）梳理出的一个应用场景是：在不同的页面状态或页面跳转之间，把同一个元素用平滑的形变效果串起来，比如让它的大小、位置、样式平滑过渡。实际场景比如，把商品列表页里的小缩略图，平滑展开成商品详情页的全屏大图。

你会注意到，这个场景描述里压根没提 View Transitions API。它只是在描述开发者想要实现的一个效果，而 View Transitions API 恰好是实现它的最佳选择。这点很重要，因为开发者一般不会跟 AI 说"请用某某 API"，他们可能根本不知道有这个功能存在。开发者提的是"我想做某个效果"，所以我们的指南必须能匹配上开发者实际会输入的提示词。

第二点，每条指南都针对 AI 做了优化。怎么说呢？就是每条指南都很精炼、直奔主题，只包含 AI 正确执行所需的最小信息量。这比让 AI 去网上搜效率高得多——就算 AI 能在网上找到最新的准确信息（这还不一定），它也得把搜到的内容全读一遍，消耗一大堆 token，就为了找出跟你需求相关的那点东西。而且说不定还找不到。我们的指南本身就按场景划好了范围，已经审核验证过准确性，而且就放在你本地设备上。

第三点，每条指南里都有一节专门写**推荐方案和降级策略**，确保这个场景的实现能在各种浏览器上都跑得通。

最后这一点我想多展开讲讲，因为这个跨浏览器降级指南，在我看来是这个工具最牛的地方。我来解释一下它是怎么工作的—— 对于每篇指南，我们首先用目前最现代的方案记录下理想的实现方式。在指南的这部分内容里，我们完全不考虑浏览器兼容性的问题。我们假设任何特性都可以使用，然后挑出最适合当前任务的那个特性来用。我们之所以这样做，是因为我们想支持那些正在尝试新特性的人、写浏览器插件的人、做 Electron 应用的人、搭建公司内部工具的人，以及任何不太需要关心跨浏览器兼容的场景。我们不希望 AI 工具在条件允许的情况下，还拦着开发者不让用新的 API。不过，如果在理想实现里用到的某些特性还没有被纳入"基线广泛可用"的范围，那我们一定会给出一个降级方案。而且只要可能，我们都会确保这个降级方案能在所有浏览器上跑得通。唯一的例外是像 WebMCP 这种——它真的没有任何跨浏览器的替代方案。但除此之外，只要存在跨浏览器的降级方案，我们就一定会用上。如果没有的话，我们也会非常清楚地标明这个特性只在部分浏览器里能用，并告诉你怎么去做特性检测。

我们之所以采用这种"先讲纯现代方案、再讲降级方案"的两层结构，是为了让 AI 编码助手能根据你网站的浏览器兼容要求，自由选择怎么来实现这个功能。

接下来就是我今天想讲的这个工具包里的最后一个特性——基线集成功能。我前面说过，我们希望编码助手能根据你的浏览器支持要求来定制实现方式。要做到这一点，有两件事得先准备好。第一，你得告诉你的 AI 助手你的基线目标，或者说你需要支持哪些浏览器。AI 有时候能从你项目里的各种配置文件猜出这些信息，但我们觉得还是明确写出来比较好，建议你直接把这段信息放到 agents.md 文件里。如果你没有声明具体的浏览器支持要求，那 AI 就会默认按"基线广泛可用"来处理。

第二，在我们每一篇指南的降级方案部分，我们都附上了详细的浏览器兼容性数据，这样 AI 就能根据你在 agents.md 里写的浏览器支持要求，判断到底需不需要用某个降级方案。举个例子，如果你在做 Electron 应用，并且在 agents.md 里写明只需要支持 Chrome 144 及以上版本，那么当 AI 遇到一个像"滚动驱动动画"这样的降级方案时——这个特性在 Chrome 115 就已经上线了——AI 就可以直接忽略这个建议，只用那个现代化的实现方式。这样一来，AI 就能针对你项目具体的浏览器支持要求，量身定制每一篇指南的实现，而不用总是走"取最小公约数"那种保守路线。

好，这就是现代 Web 指南的一个快速介绍。我觉得到这儿，可能很多人心里都在想："听起来挺不错的，但实际上到底好不好用呢？"我确实准备了一些演示，接下来会给大家看看实际效果。但是在演示之前，我想先正面回应一下这个问题，因为我觉得这很重要。我觉得作为整个行业，我们有必要去评估我们使用的这些技能到底效果怎么样。

我们在启动这个项目的时候，其中一个核心要求就是：每一个用到的特性都必须有一套严格可靠的评估标准。目前为止，我们的评估套件已经包含了 554 条独立的断言，覆盖了大部分的指南内容；我们计划在今年晚些时候的稳定版发布之前，把覆盖率扩到 100%。我们每天都会针对几款主流的 AI 编码模型跑一遍完整的评估套件，目前包括 Gemini 3.1、Claude Opus 4.7 和 GPT 5.5。

屏幕上大家看到的是所有这些编码模型所有评估结果的一个汇总可视化。右边那个绿色数字是安装了现代 Web 指南之后，AI 通过评估断言的平均通过率。左边那个数字是同一批断言，但这次是在没有安装现代 Web 指南的 AI 上跑出来的通过率。我们把这叫做"有指导 vs 无指导"的通过率。

可以看到，有指导的 AI 比没指导的 AI 表现得明显要好不少。虽然还没到 100%，但我们一直在不断地打磨我们的指南和评估，就是想把这个数字拉到尽可能高。

那这些单独的评估到底长什么样呢？其实就是给 AI 一个提示词，然后针对 AI 生成的结果跑一组断言或者测试。屏幕上这个例子就是我们的"滚动进度指示器"用例最近一次评估的内容。我们会跑屏幕上那段提示词，并配上一组断言，来测试它是不是真的按指南的要求来实现。对于这个用例，测试要检查的是：实现方案是不是正确地用了"滚动驱动动画"，而不是退而求其次地用 scroll 事件监听器。

可以看到，在这一次的运行里，有我们指南指导的情况下，所有测试都通过了；没我们指南指导的情况下，所有测试都没通过。

这个是这些模型当下的表现快照，但不难想象，以后情况会变的。随着模型越来越强、浏览器生态也在不断变化，我们完全预期：这些评估里很多项，到后面就算没有我们的指南指导，AI 也能通过了。

所以我前面说过，我们每天都会跑这些评估。我们的计划是密切观察、持续监控结果，然后一旦发现某个用例在所有主流模型上，无指导版本的表现已经稳定地跟有指导版本一样好了，就把那个用例从指南里拿掉。这样我们就能让整个工具包保持精简、节省 token 占用，把精力集中在……（未完） 关于那些不是新特性、或者是新特性但模型不了解、训练数据也没覆盖的功能。但也要记住，新特性每个月都在浏览器里上线，而知识截止日期远远落后。所以，我们相信这种指导在未来很长一段时间内都是必要的。说到未来，我想聊聊我们如何看待这个项目随时间的演进。

在这个早期预览版里，我们推出了大约十几个高级别学科指南和超过100个详细的使用案例指南。但我们肯定不会止步于此。我们的计划是继续添加更多指南，以更好地覆盖现有的 Web 平台，同时也会覆盖新上线的新特性。事实上，我们的目标是：Chrome 中发布的每一个新 Web 平台特性，在它进入 Chrome 稳定版的时候，都能被这个工具覆盖。如果你去看 chromestatus.com，你会发现 Chrome 每一个里程碑版本都会发布大量新特性。所以，这绝对是一个雄心勃勃的目标，但我们认为，在一个全新的智能体（agentic）世界里，这对支持 Web 持续健康和成功至关重要。我们目前也在和其他浏览器厂商接触，希望也能覆盖他们发布的功能特性。
