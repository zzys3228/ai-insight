# 会议演讲总结：使用 Google TPU 软件栈扩展 AI

## 核心要点

**一、AI 模型演进与算力挑战**
Google 近年来持续推出 Gemini、Gemma、Imagen、Omni 等创新模型，解锁多模态理解、推理和智能体工作流等突破性能力。构建大语言模型主要分为四个阶段：神经网络设计（JAX 框架）、预训练、后训练（监督微调 SFT + 强化学习 RL）和推理。其中，推理阶段因思考模型消耗大量 token，正成为智能的关键来源，且对算力的需求日益增长。

**二、TPU 硬件差异化设计**
针对训练和推理的不同需求，Google Cloud 推出 TPU V8 系列：**TPU 8T** 针对大规模训练工作负载优化，聚焦高吞吐量和扩展效率；**TPU 8I** 专为推理设计，主打低延迟和成本效益。

**三、核心软件工具栈**
1. **vLLM**：推理部署引擎，通过 Paged Attention（分页注意力）优化 KV 缓存管理，借助 Prefix Caching（前缀缓存）避免重复计算，并采用 Continuous Batching（连续批处理）实现 token 级动态调度，最大化硬件利用率，支持 PyTorch 和 JAX 框架，可在 GPU 与 TPU 间无缝迁移。
2. **MaxText**：基于 JAX 的大语言模型高性能参考实现，可扩展至数千芯片。
3. **Tunix**：轻量级后训练框架。
4. **Torch TPU**：让 PyTorch 代码无需修改即可运行在 TPU 上。

**四、实操演示**
演讲中展示了在 8 块 Trillium 芯片虚拟机上，通过预构建 Docker 容器部署 Gemma 4（310 亿参数）模型的过程。开发者仅需配置张量并行（设为 8）即可让模型横跨所有芯片运行，前端代码与 GPU 版本基本一致。

**五、降低使用门槛**
推理和后训练任务可借助 Kaggle 或 Colab 提供的免费 TPU 完成，大幅降低开发者试错成本。

如果你没接触过推测解码，这个想法挺疯狂的。我第一次听说的时候，反应也是：等等，这有点离谱啊。思路是这样的：如果你有一个非常大的模型，预测下一个 token 计算量很大。而推测解码的做法是，你有一个小模型，让小模型往前冲，提前预测 token，基本就是在猜大模型会说什么。出于各种原因，实际上让大模型去验证小模型的输出，比让它自己预测 token 还要快。所以这想法挺疯的。

通常你做推测解码的时候，是自回归的方式，说白了就是一次一个 token。如果你这么做，还是会有瓶颈——一个 token，下一个 token，再下一个 token……

这个 PR 做的是扩散式推测解码。不是一次预测一个 token，而是用扩散模型一次性全预测出来，这就消除了那个瓶颈。然后我们把它集成进 VLM TPU 的推理里，在一堆任务上获得了大约 3 倍的加速。
