## 📝 内容总结

## 会议演讲总结

**主题：使用 Chrome DevTools for Agents 增强 AI 编程工作流程**

**核心要点：**

1. **发布背景**：Google I/O 大会上发布的新工具 "Chrome DevTools for Agents"，通过 NPM 包形式提供，让编程 Agent 获得使用 DevTools 的能力，从而获得开发者整个职业生涯中受益的紧密反馈闭环。

2. **核心理念**：编码 Agent 虽然能胜任工作，但缺少了 DevTools 提供的"实时调试反馈循环"。通过赋予它们访问运行时数据的能力，能显著提升调试效率。

3. **实际演示**：以注册表单调试为例，传统 Agent 只能搜索代码中的错误信息"app error 1003"，而启用 DevTools for Agents 后，Agent 可在真实 Chrome 实例中打开页面、捕获网络请求、控制台日志等运行时数据，从而定位到解决方案。

4. **技术架构**：底层基于 Puppeteer/Puppeteer Core，支持两种接入方式：**MCP 服务器**和 **CLI 脚本**（后者更节省 token）。目前提供六项技能，分为"故障排查类"和"通用知识类"两大类，涵盖无障碍调试、内存分析等场景。

5. **使用方式**：用户只需修改提示词，从"查看具体文件"改为"查看运行时"，配合 source maps 即可引导 Agent 完成端到端的自动修复流程。

虽然 DevTools for agents 是个新东西，但它其实是建立在一个非常扎实且历史悠久的基础之上的。它用的是同一个团队维护了好多年的工具，也就是 Puppeteer 和 Puppeteer。我们来拆解一下。

从本质上讲，DevTools for agents 就是一个 NPM 包，你可以在任何装了较新版本 Node.js 的机器上安装它。你的编程智能体（比如 Antigravity、Gemini CLI 或者你喜欢的任何其他工具）用的就是这个包。你的智能体可以通过 MCP 服务器或者 CLI 来使用它，这两种方式都是同一个 NPM 包的一部分。用 CLI 的话，你的编程智能体可以通过脚本的方式来操作 DevTools for agents，相比通过 MCP 服务器使用会更节省 token（令牌）。

第三个、也是最后一个把这些东西串起来的关键，是一套技能（skills），它教会你的编程智能体在什么时候、怎么去使用 Chrome DevTools for agents。目前，Chrome DevTools for agents 总共提供了六个技能，大致可以分为两类。

故障排查类，还有 Chrome DevTools 以及 Chrome DevTools CLI 这些技能，能帮助你的编程智能体理解核心能力、相关概念以及一般的使用方法。

而无障碍调试、内存泄漏调试（不好意思，说错了）以及优化 LCP（最大内容绘制时间）这些技能，则包含了对应专业领域的专家知识，让你的编程智能体表现得就像那个领域的专家一样。

作为现代 Web 指南的一部分，你可能在昨天的 keynote 或者 Phil 的演讲里听说过，这个技能集合在接下来的几个月里会不断扩充，帮助你的智能体调试 Web 的最新特性。如果想了解更多关于现代 Web 指南的内容，可以在 YouTube 上找到 Phil 昨天的演讲 "解锁现代 Web 能力，融入你的 AI 编程工作流"。

好，从技能说回到整体架构。DevTools for agents 是一个 NPM 包，里面包含了技能、MCP 服务器和 CLI。我们还没讲到的是——这些东西是怎么跟 Chrome 通信的呢？答案是通过 Puppeteer。就是 Chrome 工具团队出的那个库，你以前可能用过它来跑测试、自动化一些烦人的工作流，或者爬取一些数据。

虽然你用 DevTools for agents 的时候这一层基本上是看不见的，但了解一下这个是有用的，因为归根结底，是 Puppeteer 赋予了 DevTools for agents 各种能力。在我们深入讲这些能力之前，知道 Puppeteer 是中间这一层，也能帮你更好地评估安全性，以及判断什么时候该用 Chrome DevTools for agents。

举个例子，在我们最近的一个 YouTube 教程下面，有人问："DevTools for agents 能不能访问我存在 Chrome 密码管理器里的密码？" 答案很明确：不能。DevTools for agents 没法访问你存在 Chrome 密码管理器里的密码。它并不是像你作为人类用户那样在使用 Chrome。它只是通过 Puppeteer 使用了一套经过审核和测试的工具，而 Puppeteer 本身也没有把这种能力暴露出来。

另外，DevTools for agents 默认使用的是一套独立的、匿名（无登录态）的浏览器配置文件。虽然这在很大程度上降低了一些最常见的、把浏览器访问权限交给智能体所带来的安全风险，但那句老话永远都不过时：不管听起来多无聊，你都要仔细斟酌到底要把什么访问权限交给智能体。保护好你的个人数据至关重要。

太棒了。既然我们已经完整地探索了它底层的工作原理，那就让我们来看几个它到底能做些什么的例子。

正如你刚才了解到的，DevTools for agents 的能力是从 Puppeteer 那里继承来的。但是，当你向你的编程智能体下达指令时，它并不会直接使用 Puppeteer。相反，我们把 Puppeteer 的关键能力包装成了一套不同的工具，每一件都做了清晰的描述，这样你的编程智能体就能更轻松地理解和使用。

这些工具可以分成六大类。在这个部分，我想带你过三个不同的使用场景例子，看看来自多个类别的工具是如何协同工作，来完成一个特定的用户指令的。

第一个例子就是你已经看过的那个。指令大致是这样的："去 localhost，用测试账号 test 和密码 1234 注册时，把报错的调试一下并修好。" 我们让编程智能体打开页面、填好表单，然后修好底层的错误，并且自己验证一下修复结果——所有这些都是在跑在我机器上的一个真实的 Chrome 实例里完成的。这个例子结合了输入自动化、网络和导航自动化这几类工具。

我们的合作伙伴 Cyber Agent 把这个用法推到了一个新的高度：他们给一个智能体配上了 DevTools for agents 的访问权限，让它在一个小时内自动审查了 Storybook 里 32 个组件、超过 230 个故事。想了解更多，可以在 developer.chrome.com 上找到那个案例研究。

好了，我们再看最后一个使用案例，说实话，这是我最喜欢的，因为 Chrome DevTools for agents 能让你把自己的专业能力放大。比如，假设你不太懂网页性能，Lighthouse 评分只有 57 分。或者假设你不太懂无障碍或者 SEO。那现在呢，你就可以直接跟你的编程助手说，用一个很笼统的提示词，比如"提升这个项目的性能"，让它去改进这些方面。但是用这种笼统的提示词，而且没有 DevTools for agents 的话，大概率会发生什么呢？你的编程助手就会慢慢开始疯狂消耗你的 token 预算，把你项目的所有文件都读一遍，然后只挑一些它碰巧发现的东西随便改改。到最后呢，对你网站的性能其实没啥帮助。

但是有了 DevTools for agents 就不一样了。它有专门的工具，可以跑性能追踪、分析里面包含的洞察，还可以跑无障碍、最佳实践、SEO，以及那个新的代理式浏览——也就是 Lighthouse 里的审计功能。它能真正地去测试已知的问题，而不是试图从你的源代码里去猜问题是什么。这样一来，就算你对某些领域不熟悉，你和你的编程助手也能一起改进你的网站。

比如，你可以通过让 DevTools for agents 实际测量你当前的性能，然后针对一个真实的目标去改进，并利用追踪里的性能洞察，从最重要的地方开始，来解决真正的性能问题。整个过程只需要一个提示词就行，比如"去 localhost，跑一个性能追踪，然后根据所有发现采取行动"。或者你也可以用同样的模式去改进无障碍功能，而不是让你的编程助手和底层模型只是随便加几个标签。或者你也可以用同样的模式让编程助手去搞 SEO。

虽然你刚才看到的这些例子在 MCP 服务器和 CLI 下都能用，但有些场景特别适合用 CLI，因为 CLI 还能在代理循环之外运行。这样你就可以把代理的操作保存成 shell 脚本。原理是这样的：在你用 Chrome DevTools for agents 完成调试流程之后，直接提示它"把你之前的 DevTools 操作保存到一个命名合理的脚本文件里，用 DevTools CLI 来做"。用这种提示词的时候，确保你的编程助手能访问 DevTools CLI 和相关的 skill，这些能教它怎么用 CLI。

拿之前那个在 developer.chrome.com 上验证导航的例子来说，这样就能生成一个像这样的脚本：打开页面、模拟各个视口、然后截图。唯一缺的就是对结果的实际对比，这个之前录屏里 gravity 已经帮我做过了。但如果用脚本的方式来做的话，这部分就得你自己来搞了。

好了，我们来总结一下刚才看过的所有使用案例。我们看到了一个能自动修复复杂 UI 流程的提示词，一个能通过跑性能追踪或 Lighthouse 审计来放大你专业能力的提示词，一个验证 bug 报告的，还有把提示的操作保存成脚本以便在编程助手之外重新运行的。在你自己的项目里用这些提示词的变体，就能达到你在屏幕上看到的同样的效果。想探索更多这样的使用案例以及其他工具类别的例子，可以去 developer.chrome.com 上的文档看看。

现在呢，还有一个最后的话题我想跟你一起看一下——配置。虽然 DevTools for agents 在大多数情况下开箱即用，但也有一些配置选项可以让它更贴合你的需求。要配置的话，当你把 Chrome DevTools for agents 当作 MCP 服务器用的时候，去找你的 MCP 服务配置文件。对于 Gemini CLI 来说，通常是在 `.gemini/static.json`。对于 Cloud 来说，可能在你项目的根目录的 `.mcp.json` 里。对于其他助手来说，就在类似你熟悉的位置。在这个文件里，只要往 args 数组里加一个新的条目，配上你想配置的那个选项就行了。如果你通过 CLI 用 DevTools for agents 的话，记得在提示词里明确写上你想要的选项。

比如，通过 channel 这个配置选项，你可以指定 Chrome DevTools for agents 应该连接哪个 Chrome 渠道，而不是默认的 stable。或者你也可以用配置选项来启用我们还在开发的一些实验性功能，比如通过 experimental_screencasts 启用屏幕录制的实验性支持。对于 MCP 服务器的用户来说，还有一些配置选项可以关闭某些工具类别来节省 token。

但有一个配置选项我想特别提一下，因为它对 CLI 和 MCP 服务器的用户都很有用，那就是 auto-connect（自动连接）。你可以把 auto-connect 想象成跟你的编程助手共享屏幕。因为 auto-connect 允许你的编程助手连接到一个已经在运行的 Chrome 实例，然后直接从你刚才停下的地方接着干。我们来看看这为什么有意思。

为了说明这个，我们回到最开始那个例子。我们让编程助手去打开一个特定的页面，然后用一组特定的凭证登录。我们这么做的目的是让编程助手能到达跟你已经在运行的 Chrome 里一样的页面状态。但是如果我已经看到了眼前的这个错误状态，为什么还要浪费脑细胞去写一个提示词，让编程助手也进入同样的状态呢？或者更糟糕的是，还要手动从浏览器控制台复制粘贴错误信息到…… 我的编码代理。这正是自动连接（autoconnect）的用途，而且它的运作方式相当简单。你只需要在 Chrome 浏览器里打开"检查"（inspect）工具，在侧边栏中选择"远程调试"（remote debugging）这一项——嗯哼——然后勾选"允许对此浏览器实例进行远程调试"（allow remote debugging for this browser instance）这个复选框。然后确保在你的 FCP JSON 配置里使用自动连接，或者在使用 CLI 时明确提示启用它。下次你使用编码代理时，它就会连接到你刚才一直在用的那个 Chrome，并且如果连接有问题（比如你忘了开启远程调试），它也会告诉你。

不过，现在正好是时候重复一下之前那个建议了：谨慎决定你让代理访问什么内容。因为使用自动连接时，取决于你连接的是哪个 Chrome 实例，Agents 版 DevTools 可能会使用你的个人 Chrome 配置文件，而不是默认的匿名配置文件，这样就会把你的个人数据置于风险之中。因为从技术上来说，如果连接到了我的个人 Chrome 配置文件，像"去 google.com 购物，下单 999 个彩色气球"这样的指令，有很大概率会真的执行成功，带来意想不到的开销和后果——比如一大堆气球被送到我家门口。因为在这个例子里，我可能已经登录了代理最终决定使用的那家购物网站，代理需要做的只是不停地点击"下一步"、"继续"、"结账"而已。所以，当你把自动连接和个人配置文件一起使用时，一定要谨慎考虑你给代理下的指令。
