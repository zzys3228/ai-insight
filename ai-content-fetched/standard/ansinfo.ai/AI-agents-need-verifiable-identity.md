# ANS | AI 代理的信任层

AI 代理需要可验证的身份。

随着 AI 代理开始在网站、应用程序和业务系统之间运行，每次交互都需要一种简单的方式来确认哪些代理是合法的，以及由谁运营。ANS 是一个开放标准，通过域名和 DNS 为代理分配身份。

**互联网并非为识别代理而构建。**

身份是网络运作的基础。网站有域名。个人有账户和个人资料。企业有经过验证的数字资产。如果没有明确的身份，就很难验证您正在与谁或什么进行交互。

而目前，代理没有任何标准。随着越来越多的代理上线，没有跨网站、应用程序或平台识别它们的共享方式。

如果没有标准，每次代理交互都需要重复询问相同的问题：这个代理是否如它所声称的那样？应该如何处理它？

|  | 实体 | 身份 | 验证方式 |
|---|---|---|---|
| ✓ | 🌐 网站 | 域名 | DNS + TLS 证书 |
| ✓ | 👤 个人 | 账户/个人资料 | OAuth、OpenID Connect |
| ✓ | 🏢 企业 | 已验证的资产 | Google Business、EV 证书 |
| ? | 🤖 代理 | 无标准 | 尚无共享方式 |

**网络的每个参与者都有身份标准。**

**代理目前还没有——但即将拥有。**

---

ANS 为代理提供了一种标准化的身份识别方式。

ANS 是一个开放标准，将 AI 代理与域名关联，创建一种可通过 DNS 发现的可识别且可验证的身份。

这意味着代理可以展示它代表谁或什么，其身份锚定在哪里，以及其他代理、平台和系统应如何与它交互。

ANS 没有让每个公司发明自己的代理识别方式，而是为整个生态系统提供了一个共享的起点。

### 生态系统适配

**ANS 是补充——而非竞争**

发现和信任是两个不同的问题。ANS 是可与任何发现机制配合使用的信任层。

### 标准融合正在进行中

Infoblox（DNS-AID）、GoDaddy（ANS）和 Nemethi（AID）正在积极合作，在 **_ag** 标签下开发共享的 DNS 配置。这是碎片化格局正在整合而非继续分化的第一个具体证据。对共享 SVCB 记录的整合——每种协议一条记录，携带所有参与草案的发现元数据——为正在评估是否基于 ANS 构建的任何人提供了强烈的信任信号。

| 层级 | 标准 | 来源 | 功能 | ANS 的贡献 |
|---|---|---|---|---|
| 请求层 | Web Bot Auth | Cloudflare / IETF WebBotAuth 工作组 | 每次请求的 HTTP 签名认证 | ANS 添加身份保障：该签名密钥背后是谁？ |
| 发现层 | DNS-AID | Infoblox / IETF dnsop | 基于 SVCB 的代理端点发现 | ANS 为 DNS-AID 发现的端点添加信任验证 |
| 引导层 | AID | agentcommunity.org | 基于 TXT 的最小化引导 | ANS 添加版本绑定、透明日志和渐进式验证 |
| 协议层 | A2A / MCP | Linux Foundation AAIF | 代理间的通信方式 | ANS 将协议卡片包装成信任卡片，添加身份字段和 SCITT 收据 |

---

**建立在互联网已有的命名系统之上。**

就像网站使用域名和 DNS 让人们轻松找到它们一样，ANS 将这种熟悉的模式扩展到 AI 代理，使用域名和 DNS 作为代理身份的基础。而且相关基础设施已经建成并正在使用中。

这个想法很简单：如果代理代表某个组织或个人行事，其身份应该锚定在某个域名上。
