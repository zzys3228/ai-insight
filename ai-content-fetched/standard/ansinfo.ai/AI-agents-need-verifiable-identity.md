互联网并非为识别代理而构建的。

身份是网络运行的基础。网站有域名。人有账户和个人资料。企业有经过验证的数字属性。没有明确的身份，就很难验证你正在与之交互的对象是谁或什么。

而目前，代理还没有标准。随着越来越多的代理上线，在网站、应用程序或平台之间识别它们还没有统一的方式。

没有标准，每次代理交互都需要重复询问同样的问题：这个代理是否如它所声称的那样？它应该如何被处理？

网络上每个参与者都有身份标准。

代理还没有——但即将拥有。

ANS为代理提供了识别自己的标准方式。

ANS是一个开放标准，将AI代理与域名关联，创建可通过DNS发现的可识别且可验证的身份。

这意味着代理可以展示它代表谁或什么，其身份锚定在哪里，以及其他代理、平台和系统应如何与其交互。

ANS为整个生态系统提供了一个共享的起点，而不是让每个公司自行发明识别代理的方式。

**生态系统契合度**

ANS是补充——而非竞争

发现和信任是两个不同的问题。ANS是可与任何发现机制配合使用的信任层。

**标准融合正在进行中**

Infoblox（DNS-AID）、GoDaddy（ANS）和Nemethi（AID）正在积极合作，共同开发基于_dnsdelegation标签的共享DNS配置文件。这是碎片化格局正在整合而非继续分化的第一个具体证据。对共享SVCB记录的融合——每种协议一条记录，携带所有参与草案的发现元数据——为评估是否基于ANS构建提供了强有力的信任信号。

**请求层**

Web Bot Auth

Cloudflare / IETF WebBotAuth工作组

每请求HTTP签名认证。ANS添加身份保障：签名密钥背后是谁？

**发现层**

DNS-AID

Infoblox / IETF dnsop

基于SVCB的代理端点发现。ANS为DNS-AID找到的端点添加信任验证。

**引导层**

AID

agentcommunity.org

基于TXT的最小化引导。ANS添加版本绑定、透明日志和渐进式验证。

**协议层**

A2A / MCP

Linux Foundation AAIF

代理通信方式。ANS将协议卡片包装成信任卡片，包含身份字段和SCITT收据。

建立在互联网已有的命名系统之上。

就像网站使用域名和DNS让人们更容易找到它们一样，ANS将这种熟悉的模式扩展到AI代理，使用域名和DNS作为代理身份的基础。而且相关基础设施已经建成并正在使用。

理念很简单：如果代理代表一个组织或个人行事，其身份应该锚定在一个域名上。
