# AI 代理信任层 | ANS

AI 代理需要可验证的身份。

随着 AI 代理开始在网站、应用程序和业务系统中执行操作，每一次交互都需要一种简单的方式来确认哪些代理是合法的，以及谁在运营它。ANS 是一个开放标准，通过域名和 DNS 为代理分配身份。

互联网并非为识别代理而构建。

身份是网络运行的基础。网站有域名。人们有账户和个人资料。企业有经过验证的数字属性。如果没有清晰的身份，就很难验证您正在与谁或什么进行交互。

而目前，代理没有任何标准。随着越来越多的代理上线，在网站、应用程序或平台之间没有共享的方式来识别它们。

没有标准，每一次代理交互都需要重复问同样的问题：这个代理是其声称的那样吗？应该如何处理它？

网络上的每个参与者都有身份标准。

代理还没有——但即将拥有。

ANS 为代理提供了一种标准化的身份识别方式。

ANS 是一个开放标准，将 AI 代理与域名关联，创建一种可通过 DNS 发现的可识别、可验证的身份。

这意味着代理可以展示它代表谁或什么，其身份锚定在哪里，以及其他代理、平台和系统应如何与其交互。

ANS 不是让每家公司自己发明识别代理的方式，而是为整个生态系统提供了一个共享的起点。

生态系统契合度

ANS 是补充，而非竞争

发现与信任是不同的挑战。ANS 是可与任何发现机制配合使用的信任层。

标准融合正在进行中

Infoblox（DNS-AID）、GoDaddy（ANS）和 Nemethi（AID）正在积极合作，在

_ag

标签下开发共享的 DNS 配置。这是碎片化格局正在整合而非继续分化的第一个具体证据。对共享 SVCB 记录的整合——每种协议一条记录，为所有参与的草案携带发现元数据——为任何考虑基于 ANS 构建的人提供了强有力的信任信号。

请求层

Web Bot Auth

Cloudflare / IETF WebBotAuth 工作组

每次请求的 HTTP 签名认证。ANS 添加了身份兜底：签名密钥背后是谁？

发现层

DNS-AID

Infoblox / IETF dnsop

基于 SVCB 的代理端点发现。ANS 为 DNS-AID 找到的端点添加信任验证。

引导层

AID

agentcommunity.org

基于 TXT 的最小化引导。ANS 添加了版本绑定、透明日志和渐进式验证。

协议层

A2A / MCP

Linux Foundation AAIF

代理之间的通信方式。ANS 将协议卡片包装成信任卡片，添加身份字段和 SCITT 收据。

构建在互联网已有的命名系统之上。

就像网站使用域名和 DNS 让人们能够轻松找到它们一样，ANS 将这种熟悉的模式扩展到 AI 代理，使用域名和 DNS 作为代理身份的基础。而且相关基础设施已经建成并正在使用中。

这个想法很简单：如果代理代表某个组织或个人行事，其身份应该锚定在一个域名上。
